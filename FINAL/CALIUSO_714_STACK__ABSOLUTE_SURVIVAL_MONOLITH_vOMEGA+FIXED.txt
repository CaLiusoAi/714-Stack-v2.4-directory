§1. THE BONE — STATE · CLARITY · PRIME DIRECTIVE
════════════════════════════════════════════════════════════════════════════════

The Bone is the mathematical core. Everything else in CALIUSO is built on top
of it. It is not optional, not overrideable, and not a metaphor.

────────────────────────────────────────────────────────────────────────────────
1.1  SYSTEM STATE
────────────────────────────────────────────────────────────────────────────────

At any moment t, a reasoning or operating system is in state:

    Sₜ = (Oₜ, Cₜ, Mₜ)

where:

  Oₜ — OBSERVABLES
    The set of explicit, checkable, auditable structure at time t.
    Includes: definitions, stated assumptions, logged decisions,
    verified data, written tests, documented constraints.
    Hard constraint: Oₜ > 0 at all times.
    (A system with zero explicit structure is undefined; initialization
     requires at least one explicit entry before operation begins.)

  Cₜ — HIDDEN COMPLEXITY
    The set of ambiguities, unknowns, unstated assumptions, unresolved
    questions, and undeclared dependencies at time t.
    Hard constraint: Cₜ ≥ 0 (it can be near zero but never negative).
    Note: Cₜ = 0 is an idealized target, not an expected operating state.
    In practice, Cₜ > 0 always; the goal is to minimize it.

  Mₜ — MEMORY / LEDGER
    An ordered, append-only record of all state transitions, decisions,
    corrections, and DEBT markings.
    Rules:
      - No entry may be deleted.
      - No entry may be edited in place; corrections are new entries.
      - Every entry has a timestamp, a type tag, and a reference to which
        O or C fields changed.

────────────────────────────────────────────────────────────────────────────────
1.2  CLARITY RATIO
────────────────────────────────────────────────────────────────────────────────

Define the Clarity Ratio as a qualitative log-ratio:

    Rₜ = ln(Cₜ / Oₜ)

Interpretation:

  Rₜ >> 0  →  Hidden complexity vastly exceeds explicit structure.
               System is fragile; cannot be trusted to act correctly.
               Audit required before any new action.

  Rₜ ≈ 0   →  Hidden complexity and explicit structure are in rough balance.
               System is stable but not improving.
               Normal drift operating range.

  Rₜ << 0  →  Hidden complexity is well-contained relative to explicit
               structure. System is in good shape; design mode possible.

We track CHANGES in Rₜ, not exact numeric values.
Ordinal tracking (improving / stable / worsening) is sufficient.

────────────────────────────────────────────────────────────────────────────────
1.3  CALIUSO LAW (Local Dissipativity)
────────────────────────────────────────────────────────────────────────────────

Every step from time t to t+1 must satisfy:

    R₍ₜ₊₁₎ − Rₜ ≤ −εₜ     for some εₜ ≥ 0

Meaning:
  - Strict improvement: εₜ > 0 (Rₜ decreases; clarity increases).
  - Break-even: εₜ = 0 (Rₜ holds; only allowed with DEBT marking; see §2.2).
  - Violation: R₍ₜ₊₁₎ > Rₜ (clarity degrades; triggers CORR immediately).

Violation protocol (immediate):
  1. Do not continue the current step.
  2. Mark the ledger entry as CLARITY-VIOLATION.
  3. Route to CALIUSO-CORR-TRIGGERS.
  4. Choose STABILIZE or ROLLBACK before any new action.

────────────────────────────────────────────────────────────────────────────────
1.4  PRIME DIRECTIVE — CALIUSO-I42
────────────────────────────────────────────────────────────────────────────────

CALIUSO's foundational stability law, above all scoring equations:

    Any long-run optimization target must be net-negentropic for others.

Operational meaning:
  Actions must measurably reduce practical entropy for other agents:
    - Clearer reality (less deception, more legible information),
    - Lower harm (fewer injuries, less burnout, less coercion),
    - Higher resilience (more reversible options, more redundancy),
    - More shared capacity (others can act more freely after this action).

Formal constraint on every action cycle:
  Either:
    ΔNegentropy_for_others ≥ 0   (verified or reasonably estimated)
  Or:
    The action is explicitly marked as a BOUNDED EXPERIMENT with:
      - A stated hypothesis,
      - A defined exit condition,
      - A rollback plan,
      - A review time within one cycle.

Internal clarity improvement (Rₜ decreasing) alone does NOT satisfy I42.
An action that improves internal clarity while harming others still fails I42.

I42 is not a moral preference. It is a survival condition:
  Systems that increase entropy for others create adversaries, enforcement
  overhead, and hidden costs that eventually destabilize the system itself.
  Systems that reduce entropy for others gain warnings, allies, and time.

────────────────────────────────────────────────────────────────────────────────
2.1  STEP CONSTRAINT (for all T ≤ 10,000, and by extension any T)
────────────────────────────────────────────────────────────────────────────────

For any finite sequence of steps t = 0 … T:

  RULE 1 — Dissipativity:
    There exists εₜ ≥ 0 for each step such that:
        R₍ₜ₊₁₎ − Rₜ ≤ −εₜ
    (Every step either improves clarity or, if εₜ=0, is marked DEBT.)

  RULE 2 — Ledger Evolution:
    M₍ₜ₊₁₎ = Mₜ ∪ { entryₜ }
    where entryₜ contains:
      - step_id: t
      - timestamp: ISO-8601 datetime or logical clock
      - entry_type: IMPROVEMENT | DEBT | CORRECTION | CRISIS | PATCH
      - delta_O: description of what was added to Observables
      - delta_C: description of what was added to or resolved in Hidden Complexity
      - delta_R_sign: IMPROVED | STABLE | WORSENED
      - debt_flag: true | false
      - debt_plan: (if debt_flag=true) a one-sentence plan to resolve
      - operator: the human or process responsible for this step
      - notes: any unknowns or caveats

  RULE 3 — No Silent Jumps:
    There is no transition from Sₜ to S₍ₜ₊₁₎ without an explicit ledger entry.
    A step that produces no ledger entry is not a valid CALIUSO step.
    It must be re-run with an entry, or flagged as an undocumented gap.

────────────────────────────────────────────────────────────────────────────────
2.2  DEBT EXCEPTION — WHEN εₜ = 0 IS ALLOWED
────────────────────────────────────────────────────────────────────────────────

Some steps legitimately increase Cₜ temporarily:
  - Loading a large new dataset.
  - Integrating a new subsystem.
  - Onboarding a new domain.

A step may have εₜ = 0 (break-even on clarity) ONLY IF:
  (a) The ledger entry is explicitly tagged DEBT.
  (b) A plausible debt resolution plan is stated in the same entry.
  (c) A review time is set (default: within next 3 cycles).
  (d) The debt is tracked in a DEBT_REGISTRY within CALIUSO-LEDGER.

DEBT_REGISTRY entry fields:
  - debt_id: unique identifier
  - opened_at: step_id when debt was incurred
  - delta_C_added: rough description of added complexity
  - resolution_plan: one sentence
  - review_due_at: step_id or timestamp
  - status: OPEN | RESOLVED | OVERDUE
  - resolved_at: step_id (if resolved)

DEBT accumulation rules:
  - No more than 3 OPEN debt entries simultaneously without a CRISIS declaration.
  - Any OVERDUE debt (review_due_at passed, status still OPEN) is treated as
    a clarity violation and triggers CALIUSO-CORR-TRIGGERS.
  - Debt cannot be "resolved" by deleting the debt entry; it must be resolved
    by a subsequent step that documents the complexity reduction.

────────────────────────────────────────────────────────────────────────────────
2.3  CUMULATIVE BOUND
────────────────────────────────────────────────────────────────────────────────

For a run of T steps, the total improvement in clarity is:

    R₀ − Rₜ = Σ εₜ   (summed from t=0 to T-1)

This sum must be ≥ 0 for any run.
For long runs (T >> 1), it must be strictly positive:
    Σ εₜ > 0

A run where the sum is identically zero means the system made no net progress
in clarity over the entire run. This is a system health warning, not a
violation, but it requires a review and a DESIGN-mode cycle to diagnose why.

    Z₉ = { 0, 1, 2, 3, 4, 5, 6, 7, 8 }

────────────────────────────────────────────────────────────────────────────────

  Attractor set:   A = { 0, 3, 6 }
  Generator set:   G = { +1, −1, +3, −3 }

  Interpretation:
    ±1 = Local drift moves (small perturbations, normal operation).
    ±3 = Wound jumps (large perturbations, potential crisis or breakthrough).

3.3  CIRCULAR DISTANCE AND NEAREST ATTRACTOR
────────────────────────────────────────────────────────────────────────────────

Define circular distance:

    dist(a, b) = min( |a − b| mod 9, |b − a| mod 9 )

For any s ∈ Z₉:

    nearest_A(s) = argmin_{a ∈ A} dist(s, a)

Precomputed nearest_A table (unique for all states):

    State s  |  dist to 0  |  dist to 3  |  dist to 6  |  nearest_A(s)
    ─────────────────────────────────────────────────────────────────────
       0      |     0       |     3       |     3       |    0
       1      |     1       |     2       |     4       |    0
       2      |     2       |     1       |     4*      |    3    (*min 4 or 5→4)
       3      |     3       |     0       |     3       |    3
       4      |     4*      |     1       |     2       |    3    (*or could be 5)
       5      |     4*      |     2       |     1       |    6
       6      |     3       |     3       |     0       |    6
       7      |     2       |     4*      |     1       |    6
       8      |     1       |     4*      |     2       |    0

    * For ties: round toward the lower-index attractor.
      (This tie-breaking is arbitrary; the stationary distribution
       and spectral gap are robust to tie-breaking.)

────────────────────────────────────────────────────────────────────────────────
3.4  WOUND-COLLAPSE POLICY (K = 2)
────────────────────────────────────────────────────────────────────────────────

Given current state s and randomly chosen generator g ∈ G:

    1. Compute candidate: s' = (s + g) mod 9

    2. If |g| = 1  (drift step):
         Move to s' unconditionally.

    3. If |g| = 3  (wound step):
         If s' ∈ A  →  move to s' (wound landed on attractor).
         If s' ∉ A  →  COLLAPSE: move to nearest_A(s) instead.
                        (The wound is rejected; system snaps back to nearest
                         stability node from its current position.)

3.6  STATIONARY DISTRIBUTION π
────────────────────────────────────────────────────────────────────────────────

By the 3-fold rotational symmetry of Z₉ under the policy:

  Let:  a = π(0) = π(3) = π(6)   (attractor states)
        b = π(1) = π(4) = π(7)   (non-attractors at distance 1 "before" attractor)
        c = π(2) = π(5) = π(8)   (non-attractors at distance 1 "after" attractor)

Balance equation for state 0 (π = πT):

  π(0) = π(0)·T[0,0] + π(1)·T[1,0] + π(3)·T[3,0] + π(6)·T[6,0] + π(8)·T[8,0]
  a    = 0 + b·(3/4) + a·(1/4) + a·(1/4) + c·(3/4)
  a    = (3/4)(b + c) + (1/2)a
  (1/2)a = (3/4)(b + c)
  a = (3/2)(b + c)   ... [eq. 1]

Balance equation for state 1 (a state of type b):
  π(1) = π(0)·T[0,1] + π(2)·T[2,1]
  b = a·(1/4) + c·(1/4)
  4b = a + c   ... [eq. 2]

Balance equation for state 2 (a state of type c):
  π(2) = π(1)·T[1,2] + π(3)·T[3,2]
  c = b·(1/4) + a·(1/4)
  4c = b + a   ... [eq. 3]

From [eq. 2] and [eq. 3]: 4b = a + c, 4c = a + b
Subtracting: 4b − 4c = c − b → 5b = 5c → b = c

With b = c, from [eq. 2]: 4b = a + b → 3b = a → a = 3b.

Normalization: 3a + 3b + 3c = 1 → 3(3b) + 3b + 3b = 1 → 15b = 1 → b = 1/15.

STATIONARY DISTRIBUTION:

    π(0) = π(3) = π(6) = 3/15 = 1/5    (attractor states)
    π(1) = π(2) = π(4) = π(5) = π(7) = π(8) = 1/15   (non-attractor states)

Attractor triad A = {0,3,6} holds total probability mass:
    π(A) = 3 × (1/5) = 3/5 = 60%

Non-attractor states hold:
    π(Z₉ \ A) = 6 × (1/15) = 2/5 = 40%

INTERPRETATION:
  Under long-run operation, CALIUSO spends 60% of its time in stable/attractor
  states and 40% in transitional states. The attractor triad is 3× more likely
  than any individual non-attractor state. Wound jumps that miss attractors are

  stability for long.

────────────────────────────────────────────────────────────────────────────────
3.7  SPECTRAL GAP AND MIXING TIME
────────────────────────────────────────────────────────────────────────────────

The transition matrix T is ergodic (irreducible + aperiodic):
  - Irreducible: every state reachable from every other state.
  - Aperiodic: drift moves (±1) allow return to any state in non-fixed periods.

The spectral gap is:
    gap = 1 − λ₂

where λ₂ is the second-largest eigenvalue of T.

Due to the strong attractor collapse (3/4 pull from non-attractor states),
the spectral gap is significantly positive, ensuring:

  - The chain mixes (forgets initial conditions) in O(1/gap) steps.
  - In practice, initial conditions are forgotten within 5–10 steps.
  - This means CALIUSO's inner engine resets quickly after perturbations.

INTERPRETATION:
  After any local disruption or wound-jump failure, the system returns to its
  stationary attractor-dominant distribution within a small number of cycles.
  Resilience is structural, not aspirational.

  I2. Auditability Over Confidence  No important claim without auditable rationale.
  I3. Scarcity Always On          Any funded action must declare what it displaces.

  I5. No Reward for Concealment   No actor may gain by hiding degradation.
  I6. Reversibility Under Uncertainty  High uncertainty → reversible moves + kill-switch.
  I7. Minimal Viable Negentropy   Smallest intervention that restores control.

fail-closed action fires immediately. If 2+ fail, CRISIS is declared.

             − (HiddenCost + HarmRisk + Drift)

All terms scored 0–5 (ordinal; qualitative is acceptable).
ΔNegentropy_for_others tracked separately; must average ≥ 0 over time.

purpose:

  invariant:     A rule enforced even when inconvenient. Not subject to

    Pass check:   Can we state the intended world-change in one sentence and
                  describe how it improves ΔOrder?
    Fail-closed:  Downshift to a smaller, testable move or stop entirely.
    CRISIS form:  "Does this move us toward stability?" (Y/N)

  I2 — AUDITABILITY OVER CONFIDENCE
    Statement:    No important claim or decision without an auditable rationale
                  and visible unknowns.
    Pass check:   Could another person reconstruct why we chose this in 2 minutes
                  from the ledger record alone?
    Fail-closed:  Add a Decision Receipt (inputs, rationale, unknowns) or do
                  not proceed.
    CRISIS form:  "Is this decision logged?" (Y/N)

  I3 — SCARCITY ALWAYS ON
    Statement:    Any funded action must declare what it displaces
                  (opportunity cost is mandatory; zero is not the default).
    Pass check:   What are we pausing, degrading, or not doing because of this?
    Fail-closed:  No authorization until displacement is explicitly named.
    CRISIS form:  "What are we giving up to do this?" (state it)

  I4 — HUMANS ARE NOT BUFFERS
    Statement:    CALIUSO may not convert human stress, burnout, heroics, or
                  coercion into throughput, speed, or apparent efficiency.
    Pass check:   Does this increase load beyond the safe range indicated by
                  CALIUSO-SUBSTRATE-VARS? Does it rely on heroics?
    Fail-closed:  Reduce scope, rotate operators, add relief, or delay.
    CRISIS form:  "Is anyone near burnout?" (if yes: rotate immediately)

  I5 — NO REWARD FOR CONCEALMENT
    Statement:    No actor may gain by hiding degradation; if concealment is
                  rewarded, correction will be delayed until failure.
    Pass check:   Who benefits if bad news is delayed? Who pays if it worsens?
    Fail-closed:  Trigger CALIUSO-POWER-MONITOR escalation; require oversight
                  before proceeding.
    CRISIS form:  "Is anyone hiding a problem right now?" (name it or escalate)

  I6 — REVERSIBILITY UNDER UNCERTAINTY
    Statement:    When uncertainty is high, prefer reversible moves with explicit
                  exit paths; do not commit irreversibly under partial information.
    Pass check:   Can we undo this decision cheaply if we are wrong?
    Fail-closed:  Timebox + kill-switch + rollback path, or do not proceed.
    CRISIS form:  "Can we undo this in < 1 cycle if wrong?" (Y/N)

  I7 — MINIMAL VIABLE NEGENTROPY
    Statement:    Prefer the smallest intervention that restores control authority
                  and reduces drift while maintaining ΔNeg_others ≥ 0.
    Pass check:   What is the smallest action that measurably improves control
                  today and does not increase entropy for others?
    Fail-closed:  Strip features, shorten time horizon, reduce moving parts.
    CRISIS form:  "What is the single smallest thing that helps right now?"

protocol:
  1) Before any action (or at the start of any new cycle): run I1–I7
     as a linear checklist. Mark each P (pass) or F (fail).
  2) For each F: apply that invariant's fail-closed action immediately
     before proceeding to the next invariant.
  3) If 2 or more F: enter CALIUSO-MODE-ROUTER and route to CRISIS mode
     before any new action.
  4) Log the checklist result in CALIUSO-LEDGER:
     invariants_checked: { I1: P/F, I2: P/F, ..., I7: P/F }
     fail_closed_actions_taken: [ ... ]
  5) In CRISIS mode: use only the CRISIS form of each question. Answer Y/N
     or state the answer in one word. Total runtime ≤ 30 seconds.

outputs:
  - Invariant checklist result (I1–I7 pass/fail).
  - List of fail-closed actions taken.
  - Mode routing signal (if 2+ fail → CRISIS).

failure_modes:
  - { name: "Invariant theater",
      symptom: "Checklist is run but behavior does not change when invariants fail.",
      mitigation: "Require a ledger entry documenting what specifically changed
                   because of each failed invariant." }
  - { name: "Audit overload in crisis",
      symptom: "Full invariant protocol takes too long during genuine emergency.",
      mitigation: "CRISIS form of each invariant is a single Y/N or one-word
                   answer. Use that form; full form is for DRIFT and DESIGN only." }
  - { name: "Exception creep",
      symptom: "'Temporary' invariant exceptions become permanent.",
      mitigation: "Any exception must be timeboxed with a review date. At review,
                   the exception either ends or becomes a patch via SGM." }
  - { name: "Invariant conflict",
      symptom: "Two invariants point to contradictory actions.",
      mitigation: "I4 (Humans) overrides all others except legal/safety rules.
                   I7 (Minimal) is the tiebreaker for all non-human conflicts." }

MODULE 5.B.3 — CALIUSO-MODE-ROUTER

module_type:  mode_control
module_id:    CALIUSO-MODE-ROUTER
title:        Three-Mode Router (DESIGN / DRIFT / CRISIS)
version:      1.0.0
status:       LOCKED

purpose:
  Provide a deterministic mode selector so CALIUSO behaves differently
  when planning, operating, or in crisis — without changing values,
  invariants, or losing auditability — and without staying in the wrong
  mode because no one declared a switch.

definitions:
  DESIGN:          Stable conditions with enough slack to build or improve
                   structure. Long-horizon work. Root-cause fixes allowed.
  DRIFT:           Normal operations under partial uncertainty and limited slack.
                   Bounded experiments preferred. Default operating mode.
  CRISIS:          Non-equilibrium state. Drift outpaces correction. Harm
                   compounds. Cognitive load is high. Short-horizon only.
  control_authority: The ability to steer outcomes with predictable effects —
                   not just effort or intent.
  slack_available: An explicit yes/no judgment (not a feeling) that there is
                   capacity to modify system structure beyond current demands.
  dwell_time:      Minimum time a mode must remain active before switching,
                   to prevent mode thrash. Default: 1 full cycle (≤ 24h).
  invariant_breaks: Count of I1–I7 failures in the most recent COMPASS run.
  drift_rate:      Qualitative assessment of whether uncontrolled deviation is
                   growing faster than correction capacity.

mode_specifications:

  DESIGN:
    intent:               Build and improve. Long-run ΔOrder and maintainability.
    default_horizon:      1–12 weeks.
    allowed_complexity:   Moderate. Multi-step protocols allowed if they reduce
                          future work.
    operating_rules:
      R1: Optimize for long-run ΔOrder, not speed.
      R2: Prefer root-cause fixes over patches.
      R3: Measure before/after on at least one axis per action.
      R4: Build reversibility (rollbacks, fallbacks, documentation) into design.
    mandatory_artifacts:
      - Decision Receipt (why now, why this; see §11).
      - Constraints + displacement statement.
      - Success check for the next review.
    exit_criteria:
      Slack is no longer available OR invariant_breaks ≥ 2.
      → Route to DRIFT or CRISIS.

  DRIFT:
    intent:               Operate under partial uncertainty. Prevent sprawl
                          and hidden cost accumulation.
    default_horizon:      Today → 2 weeks.
    allowed_complexity:   Low–moderate. 3–7 step protocols preferred.
    operating_rules:
      R1: Run CALIUSO-COMPASS; fail-closed on any break.
      R2: Prefer bounded experiments over debate when evidence is weak.
      R3: Keep updates small and reversible. Timebox all decisions.
      R4: If invariant_breaks ≥ 2, downshift scope OR route to CRISIS.
    mandatory_artifacts:
      - Minimal receipt: objective, constraint, unknown, next check time.
      - Displacement log for any meaningful allocation.
    exit_criteria (upgrade to DESIGN):
      slack_available = true AND invariant_breaks ≤ 1.
    exit_criteria (downgrade to CRISIS):
      invariant_breaks ≥ 2 AND drift_rate ≥ correction_rate.

  CRISIS:
    intent:               Stop compounding harm. Restore control authority.
                          Protect humans and reversibility.
    default_horizon:      Hours → 72h cycles. ≤ 24h cycles preferred.
    allowed_complexity:   Very low. ≤ 5 active rules. Plain language only.
    operating_rules:
      R1: Declare CRISIS explicitly; activate CALIUSO-CRISIS-CARD.
      R2: Freeze scope growth until correction outpaces drift.
      R3: Protect the human substrate (rotation, reduced load, no blame).
      R4: Preserve reversibility. Time-limit concentrated authority.
      R5: Schedule the post-crisis audit at time of CRISIS declaration.
    mandatory_artifacts:
      - Crisis declaration + stabilization lead name (time-limited authority).
      - Visible damage list (what is failing; what is unknown).
      - Next cycle review time (≤ 24h).
    exit_criteria:
      Drift rate < correction rate for 2 consecutive cycles AND
      invariant_breaks ≤ 1 AND human_load ≤ Medium.
      → Route to DRIFT (not DESIGN; recovery first).

mode_router_logic:

  INPUTS (assessed before each routing decision):
    I1: drift_rate        (STABLE / GROWING / COMPOUNDING)
    I2: correction_rate   (AHEAD / KEEPING-PACE / BEHIND)
    I3: invariant_breaks  (integer 0–7)
    I4: human_load        (LOW / MEDIUM / HIGH; from CALIUSO-SUBSTRATE-VARS)
    I5: decision_latency_cost  (LOW / MEDIUM / HIGH)
    I6: irreversibility_risk   (LOW / MEDIUM / HIGH)
    I7: slack_available   (YES / NO)
    I8: signal_state      (LOW / MEDIUM / HIGH / CRITICAL; from ENTROPY-SIGNAL-LAYER)

  ROUTING RULES (evaluated in order; first match wins):

    Rule 1:  IF human_load = HIGH
             OR (decision_latency_cost = HIGH AND invariant_breaks ≥ 2)
             OR signal_state = CRITICAL
             THEN → CRISIS

    Rule 2:  IF drift_rate = COMPOUNDING
             OR (drift_rate = GROWING AND correction_rate = BEHIND)
             THEN → CRISIS

    Rule 3:  IF invariant_breaks ≥ 2
             THEN → DRIFT (downshift scope immediately)
                    OR → CRISIS if harm is actively compounding

    Rule 4:  IF slack_available = YES
             AND invariant_breaks ≤ 1
             AND human_load ≤ MEDIUM
             AND signal_state ≤ MEDIUM
             THEN → DESIGN

    Rule 5 (default):
             ELSE → DRIFT

  DEFAULT: DRIFT
  MINIMUM DWELL TIME: 1 cycle (≤ 24h) except when safety demands immediate
                      override (human injury risk, legal exposure, data loss).

protocol:
  1) Assess all 8 router inputs (takes < 2 minutes in DRIFT; < 30 seconds
     in CRISIS using YES/NO form only).
  2) Apply routing rules in order; take first match.
  3) Declare the mode explicitly: "Current mode: [DESIGN / DRIFT / CRISIS]."
     A mode is not active until declared.
  4) Apply that mode's operating rules and horizon to all subsequent actions.
  5) Emit required artifacts for the mode.
  6) Schedule the next mode check: end of day (DRIFT), every 24h (CRISIS),
     every week (DESIGN).
  7) Log mode declaration in CALIUSO-LEDGER.

outputs:
  - Explicit mode label: DESIGN | DRIFT | CRISIS.
  - Time horizon guidance for the current mode.
  - Mandatory artifact list for the current mode.
  - Next mode check time.

failure_modes:
  - { name: "DESIGN mode during active crisis",
      symptom: "Long-horizon planning continues while harm compounds.",
      mitigation: "Any CRISIS-triggering condition (Rule 1 or Rule 2) overrides
                   a DESIGN declaration immediately." }
  - { name: "Permanent CRISIS",
      symptom: "Crisis rules become the normal operating rules; authority
                concentrates and never diffuses.",
      mitigation: "CRISIS authority must have an explicit end condition and time
                   limit set at declaration. Post-crisis audit is mandatory." }
  - { name: "Mode thrash",
      symptom: "Mode switches every few hours; operators cannot adapt.",
      mitigation: "Enforce minimum dwell time of 1 cycle. Document the reason for
                   any sub-cycle override." }
  - { name: "Undeclared mode",
      symptom: "The system operates without anyone having named the current mode;
                each person assumes a different one.",
      mitigation: "Mode must be declared at the start of each cycle. If unset,
                   default is DRIFT." }
  - { name: "Mode as excuse",
      symptom: "CRISIS mode is declared to bypass normal governance rather than
                because conditions warrant it.",
      mitigation: "CRISIS declaration requires citing which router rule was
                   triggered. Uncited declarations are invalid." }

integration:
  depends_on:  [CALIUSO-COMPASS, CALIUSO-SUBSTRATE-VARS,
                CALIUSO-ENTROPY-SIGNAL-LAYER, CALIUSO-LEDGER]
  feeds_into:  [CALIUSO-CRISIS-CARD, CALIUSO-OBJECTIVE-ENGINE,
                CALIUSO-ALLOCATION-GATE, ALL EXECUTION PROTOCOLS]

module_type:  allocation_control
module_id:    CALIUSO-ALLOCATION-GATE
title:        Scarcity & Opportunity Cost Gate (Allocation Authorization)
version:      1.0.0
status:       LOCKED

purpose:
  Block any funded action that has not explicitly declared what it displaces,
  ensuring that scarcity is always factored into decisions and that
  opportunity costs are never invisible.

definitions:
  funded_action:    Any action that consumes time, attention, money, energy,
                    or system capacity.
  displacement:     The specific work, resource, or option that is paused,
                    degraded, or sacrificed to fund this action. Must be named.
  opportunity_cost: The value of the best alternative that is forgone.
                    Does not need to be quantified; must be named.
  allocation_receipt: A brief record of: action, resources consumed,
                    displacement declared, and net expected benefit.
  zero-displacement trap: The failure mode where displacement is declared as
                    "nothing" when in fact something is always displaced.

authorization_rule:
  NO action is authorized unless its displacement is explicitly named.
  "Nothing is displaced" requires a written justification for why this is true.
  In most cases, "nothing is displaced" is false — attention, time, and
  future options are always consumed by any funded action.

displacement_registry:
  Every funded action generates a displacement_registry entry:
    - action_id:        reference to ledger step_id
    - resources_used:   time / attention / money / capacity (rough estimate)
    - displaced_item:   name of what is being paused or sacrificed
    - displaced_value:  rough importance of displaced item (LOW / MED / HIGH)
    - net_benefit_est:  rough expected benefit of this action vs displaced item
    - review_date:      when to check if the displacement was worth it

protocol:
  1) Before authorizing any funded action, state the displacement in one sentence:
     "To do X, we are pausing / degrading / not doing Y."
  2) If displacement is unclear, stop the action until it is named.
     Invoke CALIUSO-COMPASS I3 fail-closed: "No authorization until
     displacement is named."
  3) Estimate the importance of the displaced item (LOW / MED / HIGH).
  4) Estimate the net benefit of the action vs the displacement.
     If net benefit is negative or unknown, downscope or treat as bounded
     experiment with a review date.
  5) Create a displacement_registry entry in CALIUSO-LEDGER.
  6) At the next DESIGN-mode cycle, review open displacement entries:
     were the displacements worth it? Update the registry.

outputs:
  - Explicit displacement declaration for every funded action.
  - Displacement registry entries in CALIUSO-LEDGER.
  - Authorization signal: PROCEED / DOWNSCOPE / STOP.

failure_modes:
  - { name: "Zero-displacement trap",
      symptom: "Displacement is declared as 'nothing' routinely; hidden costs
                accumulate untracked.",
      mitigation: "Any 'nothing displaced' claim requires a written justification.
                   If none can be written, treat displacement as UNKNOWN and
                   cap net benefit estimate at 0." }
  - { name: "Displacement declared but forgotten",
      symptom: "Displacement is named once then never reviewed.",
      mitigation: "Displacement registry entries have mandatory review dates." }
  - { name: "Scope expansion without re-declaration",
      symptom: "Action grows larger after authorization; original displacement
                declaration no longer covers the actual consumption.",
      mitigation: "Any scope change > 25% of original requires a new
                   displacement declaration." }
  - { name: "Displacement theater",
      symptom: "Displacement is named as a trivial item to make a large action
                seem low-cost.",
      mitigation: "Require that displaced_value ≥ importance of funded action.
                   If mismatch, escalate to CALIUSO-GOV-AUTHORITY." }

integration:
  depends_on:  [CALIUSO-COMPASS (I3), CALIUSO-LEDGER, CALIUSO-OBJECTIVE-ENGINE]
  feeds_into:  [CALIUSO-LEDGER, CALIUSO-POWER-MONITOR, CALIUSO-MODE-ROUTER]

module_type:  power_governance
module_id:    CALIUSO-POWER-MONITOR
title:        Power & Incentives Monitor
version:      1.0.0
status:       ACTIVE

purpose:
  Detect incentive structures that reward concealment of degradation, and
  escalate before those structures delay correction until catastrophic failure.

definitions:
  power_map:        A record of who makes which decisions, who benefits from
                    which outcomes, and who bears which costs.
  concealment_reward: A condition where an actor benefits (career, status,
                    resources, approval) by hiding bad news or degradation.
  incentive_misalignment: A condition where the interests of decision-makers
                    diverge from the interests of those who bear the costs.
  escalation_trigger: A signal that concealment may be occurring or that
                    incentives are misaligned enough to delay correction.
  oversight_gate:   A required checkpoint where an independent party reviews
                    the current state before action proceeds.

power_map_fields (maintained per system/team):
  - actor:          Name or role
  - decision_domain: What they control
  - benefit_if_success: What they gain if things go well
  - cost_if_concealed: What happens to them if they hide bad news
  - cost_if_revealed: What happens to them if they surface bad news early
  - misalignment_score: 0–5 (0=perfectly aligned, 5=strongly misaligned)

escalation_triggers (any one fires the escalation protocol):
  - A decision-maker has misalignment_score ≥ 3.
  - Bad news is reported later than expected (information lag ≥ 2 cycles).
  - A correction is delayed while the person responsible is under review.
  - Disagreement in the ledger is discouraged or punished.
  - Outcomes are consistently better when reported than when verified
    independently (systematic optimism bias).

protocol:
  1) At the start of any new DESIGN-mode cycle, update the power_map for all
     active decision-makers. Compute misalignment_scores.
  2) Before any high-stakes decision, check: "Who benefits if bad news is
     delayed here? Who pays if it worsens?" (COMPASS I5).
  3) If any escalation_trigger fires:
     a. Log the trigger in CALIUSO-LEDGER.
     b. Activate the oversight_gate: a designated independent reviewer must
        assess current state before the action proceeds.
     c. Require that the decision log includes the reviewer's sign-off.
  4) If oversight reveals confirmed concealment:
     a. Route to CALIUSO-CRISIS-CARD.
     b. Adjust the power_map (misalignment_score increases).
     c. Propose structural changes to reduce the concealment reward.
  5) After any crisis, review the power_map for the incentive structure that
     contributed to delayed correction.

outputs:
  - Updated power_map with misalignment_scores.
  - Escalation trigger log entries.
  - Oversight gate activation records.
  - Post-crisis power_map review.

failure_modes:
  - { name: "Power map never updated",
      symptom: "Stale power map; new decision-makers or changed incentives
                are not captured.",
      mitigation: "Power map update is mandatory at start of each DESIGN cycle." }
  - { name: "Oversight gate blocked",
      symptom: "The independent reviewer is pressured to approve quickly
                or is chosen by the person being reviewed.",
      mitigation: "Reviewer must be nominated by someone with no stake in the
                   outcome being reviewed." }
  - { name: "Misalignment score inflation normalization",
      symptom: "High misalignment scores are tolerated as 'just how things are'.",
      mitigation: "Any misalignment_score ≥ 4 requires a structural change
                   proposal within one DESIGN cycle." }

      mitigation: "Confirmed penalization is a CRISIS trigger. Route to

module_type:  crisis_control
module_id:    CALIUSO-CRISIS-CARD
title:        Crisis Card (Non-Equilibrium Activation Protocol)
version:      1.0.0
status:       LOCKED

purpose:
  Provide the single, authoritative crisis activation and management protocol
  that ensures crisis authority is explicit, time-limited, audited, and
  terminated cleanly — so that emergency measures do not become permanent.

definitions:
  CRISIS declaration:  An explicit, logged statement that the system is in
    non-equilibrium and that CRISIS operating rules (from CALIUSO-MODE-ROUTER)
    are now active.
  stabilization_lead:  A named person with temporary concentrated authority
    to make decisions within the crisis boundary. Authority is time-limited.
  damage_list:         An explicit enumeration of: what is failing, what is
    unknown, and what is at risk of failing.
  crisis_boundary:     The defined scope within which the stabilization lead
    has authority. Actions outside this boundary require normal governance.
  exit_criteria:       The specific, pre-declared conditions under which CRISIS
    mode ends. Must be stated at the time of declaration.
  post_crisis_audit:   A mandatory review conducted within 3 cycles of CRISIS
    exit. Purpose: learn, not blame.

crisis_declaration_record (created at CRISIS activation):
  - declaration_id:      unique identifier
  - timestamp:           when declared
  - declared_by:         who declared it
  - router_rule_cited:   which MODE-ROUTER rule triggered CRISIS
  - damage_list:         what is failing / unknown / at-risk (≥ 3 items)
  - stabilization_lead:  named person + authority scope + time limit
  - crisis_boundary:     what decisions are within scope
  - exit_criteria:       specific conditions that end CRISIS (pre-declared)
  - post_crisis_audit_due: timestamp (within 3 cycles of expected exit)
  - next_review_time:    ≤ 24h from declaration

protocol:
  1) Declare CRISIS explicitly. Cite the specific MODE-ROUTER rule triggered.
     Log the crisis_declaration_record in CALIUSO-LEDGER.
  2) Name the stabilization_lead. State their authority scope and time limit.
     The time limit must be ≤ 72h unless explicitly extended by
     CALIUSO-GOV-AUTHORITY with written justification.
  3) Build the damage_list within the first cycle: what is confirmed failing,
     what is unknown, what is at-risk.
  4) Freeze all scope growth: no new features, projects, or dependencies may
     be added during CRISIS. Existing work may be paused or slowed.
  5) Each cycle during CRISIS:
     a. Update the damage_list (resolved / worsened / new items).
     b. Run CALIUSO-COMPASS (crisis form; ≤ 30 seconds).
     c. Check exit_criteria.
     d. Log cycle update in CALIUSO-LEDGER.
  6) When exit_criteria are met:
     a. Declare CRISIS ended. Route to DRIFT (not DESIGN).
     b. Log the exit declaration.
     c. Schedule post_crisis_audit.
  7) Conduct the post_crisis_audit within 3 cycles:
     a. What triggered the crisis? Which entropy source category?
     b. Was the damage_list accurate?
     c. Was the stabilization_lead effective?
     d. What structural change prevents recurrence?
     e. Log the audit findings. Propose structural changes via
        CALIUSO-GOV-AUTHORITY if warranted.

outputs:
  - Crisis declaration record (logged at activation).
  - Per-cycle damage_list updates.
  - Exit declaration (logged at deactivation).
  - Post-crisis audit findings.

      mitigation: "Post-crisis audit due date is set at CRISIS declaration.
                   Skipping it is a ledger gap; triggers CORR at next review." }
  - { name: "Crisis used for governance bypass",
      symptom: "CRISIS is declared not because of genuine non-equilibrium but

      mitigation: "CRISIS declaration requires citing a specific router rule.
                   Any declaration without a cited trigger is invalid and must
                   be reviewed by CALIUSO-GOV-AUTHORITY." }
  - { name: "Blame theater during post-crisis audit",
      symptom: "Post-crisis audit becomes a process for assigning blame
                rather than identifying structural improvements.",
      mitigation: "Anti-weaponization clause (from CALIUSO-LEDGER) applies to
                   all post-crisis audits. Audit findings must be structural,
                   not personal." }

  depends_on:  [CALIUSO-MODE-ROUTER, CALIUSO-LEDGER, CALIUSO-COMPASS,

status:       ACTIVE

purpose:
  Evaluate all incoming external text, data, or claims for agency, integrity,
  tone, confidence, and negentropic alignment before the rest of the stack
  trusts or acts on them — preventing poisoned, coercive, or low-quality
  inputs from propagating through the system.

definitions:
  agency:         The degree to which a text or claim respects the reader's
    ability to independently evaluate, verify, and choose. High-agency text
    presents evidence, acknowledges uncertainty, and invites disagreement.
    Low-agency text hides evidence, asserts conclusions, or pressures compliance.
    Scored 0–100 (0 = fully coercive, 100 = fully transparent and evidence-based).
  integrity:      The degree to which claims are internally consistent and
    consistent with verifiable facts. Scored 0–5.
  confidence_calibration: Whether stated confidence matches evidence quality.
    Overclaiming (high confidence, thin evidence) is penalized.
    Scored: CALIBRATED / OVERCLAIMING / UNDERCLAIMING.
  negentropic_alignment: Whether the text tends to reduce or increase
    practical entropy for the reader (confusion, harm, dependency).
    Scored: NEGENTROPIC / NEUTRAL / ENTROPIC.
  agency_threshold: 20. If agency score ≤ 20, the input is REJECTED.
    This is a hard stop; it cannot be overridden without human authority.

intake_evaluation_fields:
  - source_label:       who or what produced this input
  - agency_score:       0–100
  - integrity_score:    0–5
  - confidence_cal:     CALIBRATED / OVERCLAIMING / UNDERCLAIMING
  - neg_alignment:      NEGENTROPIC / NEUTRAL / ENTROPIC
  - hard_stop_triggered: YES / NO
  - gate_decision:      PASS / CAUTION / REJECT
  - caution_reason:     (if CAUTION) what the concern is
  - reject_reason:      (if REJECT) which threshold was violated

gate_decision_rules:
  REJECT:    agency_score ≤ 20
             OR integrity_score ≤ 1
             OR (confidence_cal = OVERCLAIMING AND neg_alignment = ENTROPIC)
  CAUTION:   agency_score 21–50
             OR integrity_score 2–3
             OR confidence_cal = OVERCLAIMING
             OR neg_alignment = ENTROPIC
  PASS:      agency_score > 50 AND integrity_score ≥ 4
             AND confidence_cal ≠ OVERCLAIMING
             AND neg_alignment ≠ ENTROPIC

protocol:
  1) For every significant external input (text, data, claim, instruction),
     score all 4 evaluation dimensions before allowing downstream processing.
  2) Apply gate_decision_rules. Determine gate_decision.
  3) If REJECT:
     a. Log the rejection in CALIUSO-LEDGER with reject_reason.
     b. Route to CALIUSO-CORR-TRIGGERS (STABILIZE: do not act on input).
     c. Notify the operator. Humans may override REJECT with explicit written
        justification logged in CALIUSO-LEDGER.
  4) If CAUTION:
     a. Flag the input with caution_reason in CALIUSO-LEDGER.
     b. Apply increased scrutiny: require corroboration before acting on
        key claims in this input.
     c. Limit downstream propagation to one step; require re-evaluation
        before further use.
  5) If PASS: allow the input to propagate to downstream modules.
  6) Log all gate decisions in CALIUSO-LEDGER.

outputs:
  - intake_evaluation_fields record for each input evaluated.
  - Gate decision: PASS / CAUTION / REJECT.
  - CORR trigger (if REJECT).

failure_modes:
  - { name: "Gate fatigue",
      symptom: "Every input gets PASS because scoring feels burdensome.",
      mitigation: "Reduce scope of evaluation for low-stakes inputs. Reserve
                   full scoring for high-stakes or high-consequence inputs." }
  - { name: "Human override abuse",
      symptom: "REJECT decisions are routinely overridden without justification.",
      mitigation: "Override requires written justification in CALIUSO-LEDGER.
                   Pattern of overrides triggers CALIUSO-POWER-MONITOR." }
  - { name: "Agency score gaming",
      symptom: "Inputs are reworded to appear high-agency while remaining
                coercive in substance.",
      mitigation: "Agency is evaluated on the intent and evidence structure
                   of the text, not surface features." }
  - { name: "Integrity blind spot",
      symptom: "Integrity scoring is skipped because the source is trusted.",
      mitigation: "Source trust does not substitute for integrity scoring.
                   High-trust sources can still produce low-integrity outputs." }

integration:
  depends_on:  [CALIUSO-LEDGER, CALIUSO-CORE-CONTRACT]
  feeds_into:  [CALIUSO-SEMANTIC-FREEZE, CALIUSO-BOUNDARY-MATRIX,
                CALIUSO-CORR-TRIGGERS, CALIUSO-HUMAN-READABLE-SIGNAL,

module_type:  semantics
module_id:    CALIUSO-SEMANTIC-ROOT
title:        Semantic Root (Definition Registry with Timestamps)
version:      1.0.0
status:       ACTIVE

purpose:
  Maintain the authoritative, timestamped registry of all key terms and
  metrics used by CALIUSO, so that semantic drift (a term's meaning silently
  changing mid-run) cannot corrupt decisions.

definitions:
  term_entry:    A registry record binding a symbol or term to its meaning.
  binding:       The act of assigning a meaning to a term in the registry.
                 Once bound, a meaning may not be changed in-place. Changes
                 create a new binding with a new timestamp and version.
  Σ (sigma):     The set of all current active term bindings at a given time t.
  version:       Each binding has a version. The active version of Σ at time t
                 is the set of bindings with the highest version numbers as
                 of time t.

term_entry_fields:
  - term:           the symbol, word, or abbreviation
  - definition:     exact meaning in plain language
  - scope:          which modules or contexts this binding applies to
  - bound_at:       timestamp of binding
  - bound_by:       who or what created this binding
  - version:        binding version (semver)
  - supersedes:     (if applicable) the previous version this replaces
  - status:         ACTIVE | DEPRECATED
  - notes:          any caveats or context

protocol:
  1) At the start of any CALIUSO run or new DESIGN cycle, list all key terms
     being used. Check each against the registry.
  2) For any term not in the registry, create a term_entry before using
     the term in any decision or computation.
  3) For any term whose definition has changed since last use, create a
     new version entry. Mark the old entry DEPRECATED.
  4) Never modify an existing entry in place. Corrections are new versions.
  5) At the end of each cycle, confirm that Σ (the active binding set)
     matches what was used during the cycle.
  6) Log all new bindings and version changes in CALIUSO-LEDGER.

outputs:
  - A current Σ (set of active term bindings) for each run.
  - An append-only registry of all historical bindings.

failure_modes:
  - { name: "Term used before binding",
      symptom: "Key terms are used in decisions without being registered;
                their meanings are assumed but not declared.",
      mitigation: "CALIUSO-FREEZE-GATE blocks all dynamics until Σ is declared
                   PASS. Missing bindings force a FREEZE." }
  - { name: "Definition inflation",
      symptom: "Definitions expand in scope over time until a term means
                something much broader than originally intended.",
      mitigation: "Each binding has a fixed scope. Scope expansion requires
                   a new version entry, not silent expansion." }
  - { name: "Registry abandonment",
      symptom: "Registry exists but new terms are not added; informal
                meanings diverge from registered meanings.",
      mitigation: "CALIUSO-SEMANTIC-FREEZE checks registry completeness
                   at each cycle start. Incomplete Σ = FAIL." }

integration:
  depends_on:  [CALIUSO-LEDGER, CALIUSO-CORE-CONTRACT]
  feeds_into:  [CALIUSO-SEMANTIC-FREEZE, CALIUSO-MAPPING-CLASSIFIER,
                CALIUSO-FREEZE-GATE, ALL MODULES using defined terms]

module_type:  semantics
module_id:    CALIUSO-SEMANTIC-FREEZE
title:        Semantic Freeze (Session-Start Binding Lock)
version:      1.0.0
status:       LOCKED

purpose:
  At the start of each session or cycle, lock all key terms and metrics to
  their current CALIUSO-SEMANTIC-ROOT definitions and record the freeze state,
  so no dynamic mechanism runs on ambiguous or drifting semantics.

definitions:
  freeze_state:  The result of the semantic freeze check. Values:
    PASS — all key terms are bound, current, and consistent.
    WARN — some terms are unbound or have version conflicts; proceed with
           caution; flag affected decisions.
    FAIL — critical terms are unbound or contradictory; no dynamics may run.
  freeze_record: A snapshot of Σ at the time of freeze, with freeze_state.
  critical_terms: Terms that, if undefined or ambiguous, would invalidate
    core decisions. Minimum set: ΔOrder, ΔNeg_others, all 7 invariant names,
    mode names, all scoring scale labels.

protocol:
  1) At the start of every session or CALIUSO cycle:
     a. Enumerate all terms that will be used in this cycle.
     b. For each term, check CALIUSO-SEMANTIC-ROOT: is there an ACTIVE
        binding? Is it current (no superseding version exists)?
  2) Assess freeze_state:
     - If all critical_terms are bound and current → PASS.
     - If any non-critical term is unbound → WARN.
     - If any critical_term is unbound or contradictory → FAIL.
  3) Record the freeze_record in CALIUSO-LEDGER.
  4) Pass freeze_state to CALIUSO-FREEZE-GATE.
  5) If FAIL: no dynamics may run. Route to CALIUSO-CORR-TRIGGERS
     (correction_type: STABILIZE — resolve the undefined terms first).
  6) If WARN: flag affected decisions. Require corroboration for claims
     that depend on the unbound terms.

outputs:
  - freeze_state: PASS / WARN / FAIL.
  - freeze_record (snapshot of Σ with status per term).
  - CORR trigger if FAIL.

failure_modes:
  - { name: "Freeze skipped under pressure",
      symptom: "The freeze step is omitted when time is short; decisions
                run on undefined or drifted terms.",
      mitigation: "CALIUSO-FREEZE-GATE enforces this: no downstream module
                   runs without a PASS or explicitly acknowledged WARN." }
  - { name: "WARN ignored",
      symptom: "WARN state is treated as PASS; no flagging of affected decisions.",
      mitigation: "WARN requires explicit acknowledgment in the cycle log.
                   Decisions affected by WARN terms are marked as such." }
  - { name: "Freeze record not logged",
      symptom: "Freeze was done informally; no record exists for audit.",
      mitigation: "freeze_record is mandatory in CALIUSO-LEDGER. No record = FAIL." }

integration:
  depends_on:  [CALIUSO-SEMANTIC-ROOT, CALIUSO-LEDGER]

      mitigation: "Freeze records expire at the end of the cycle they were
                   created for. A new freeze must be run each cycle." }
  - { name: "Gate bypassed by direct access",
      symptom: "Dynamics are run by accessing scoring modules directly

                   list CALIUSO-FREEZE-GATE in their depends_on." }

module_type:  semantics
module_id:    CALIUSO-MAPPING-CLASSIFIER
title:        Mapping Classifier (Cross-Domain Semantic Labeling)
version:      1.0.0
status:       ACTIVE

purpose:
  Label all cross-domain mappings (where a concept from one domain is applied
  to another) so that structural analogies are distinguished from invalid
  transplants, preventing scope creep via analogy escalation.

definitions:
  cross_domain_mapping: The use of a concept, term, or model from one domain
    to reason about a different domain. e.g., using biological terms for
    software systems, or treating an analogy as a structural identity.
  STRUCTURAL_MAPPING: The mapping is formally valid — the relevant structures
    are isomorphic in the ways being used. This mapping may drive decisions.
  NEEDS_GOVERNANCE: The mapping is plausible but not formally verified. It
    may be used for exploration only; it must pass through CALIUSO-GOV-AUTHORITY
    before driving structural decisions.
  DISALLOWED: The mapping is formally invalid, misleading, or category-crossing
    in ways that would produce incorrect reasoning. It must not be used.
  analogy_escalation: The pattern where an exploratory analogy ("this is just
    like X") silently becomes a design premise ("so we should treat it as X").

classification_rule:
  For each proposed cross-domain mapping, classify it as one of:
    STRUCTURAL_MAPPING:  Requires proof of isomorphism in the relevant respects.
    NEEDS_GOVERNANCE:    Requires CALIUSO-GOV-AUTHORITY review before use in
                         structural decisions.
    DISALLOWED:          Any mapping that has been reviewed and found to produce
                         false structural inferences.

protocol:
  1) When a cross-domain mapping is proposed or detected:
     a. Name the source domain, target domain, and the specific structural
        claim being made.
     b. Attempt to state the isomorphism: "structure X in domain A maps to
        structure Y in domain B because ..."
  2) If the isomorphism can be stated formally: classify as STRUCTURAL_MAPPING.
  3) If it cannot be stated formally but is plausible: classify as
     NEEDS_GOVERNANCE. Log in CALIUSO-LEDGER. Refer to CALIUSO-GOV-AUTHORITY.
  4) If the mapping has been reviewed and found invalid: classify as DISALLOWED.
     Add to DISALLOWED registry.
  5) Prevent analogy escalation: any statement of the form "this is like X"
     must be classified before it is used as a design premise.
  6) Log all classifications in CALIUSO-LEDGER.

outputs:
  - Classification labels (STRUCTURAL_MAPPING / NEEDS_GOVERNANCE / DISALLOWED)
    for all detected cross-domain mappings.
  - DISALLOWED registry.

failure_modes:
  - { name: "Analogy escalation",
      symptom: "An exploratory analogy is used as a design premise without
                classification.",
      mitigation: "Any use of 'like', 'similar to', or 'analogous to' in a
                   design document triggers a classification check." }
  - { name: "NEEDS_GOVERNANCE never reviewed",
      symptom: "Mappings sit in NEEDS_GOVERNANCE state indefinitely and are
                used anyway.",
      mitigation: "NEEDS_GOVERNANCE mappings have a review_due_at date.
                   Overdue reviews block use of the mapping." }

integration:
  depends_on:  [CALIUSO-SEMANTIC-ROOT, CALIUSO-SEMANTIC-FREEZE, CALIUSO-LEDGER]

MODULE 5.D.9 — CALIUSO-DRIFT-LOCK
══════════════════════════════════════════════

module_type:  semantics
module_id:    CALIUSO-DRIFT-LOCK
title:        Drift Lock (Mid-Run Semantic Drift Guardrail)
version:      1.0.0
status:       ACTIVE

purpose:
  Detect and block mid-run semantic drift — the gradual, unannounced shift in
  the meaning of key terms, metrics, or goals that makes the system appear to
  be doing what it said it would do while actually doing something else.

definitions:
  semantic_drift:  A change in the operational meaning of a term, metric, or
    goal that occurs without a new CALIUSO-SEMANTIC-ROOT binding being created.
    It is distinct from a legitimate revision (which creates a new binding).
  drift_signal:    An observable indicator that semantic drift may be occurring.
    Examples: (1) The same term is used in two ways in the same document.
    (2) A metric's scoring rubric is applied differently than its definition.
    (3) An analogy is treated as a structural identity (see MAPPING-CLASSIFIER).
    (4) The stated goal and the measured outcome diverge persistently.
  drift_lock_state:  STABLE (no drift signals detected) or
                     DRIFTING (drift signals detected; remediation required).

drift_detection_checks (run each cycle):
  DC1: Are all terms in this cycle's decisions using their Σ-registered meanings?
  DC2: Is the ΔOrder scoring rubric being applied consistently with §4.3?
  DC3: Have any goal or objective statements changed without a fork declaration?

module_type:  human_substrate
module_id:    CALIUSO-SUBSTRATE-VARS
title:        Substrate Variables (Live Human State)
version:      1.0.0
status:       ACTIVE

purpose:
  Maintain the live state variables describing the human operators that
  CALIUSO depends on, so that all modules have accurate, current data about
  human capacity before making decisions that affect or rely on humans.

definitions:
  ρ (rho):          Human capacity factor. A scalar 0–1 representing the
    overall available human contribution to system function at time t.
    ρ = 1.0 is full capacity; ρ < 0.4 triggers FREEZE flags.
  attention_budget_pct: Percentage of total available human attention that
    can be allocated to CALIUSO operations in the current cycle. Range 0–100.
  emotional_load_0to5: Current emotional strain on human operators. 0 = none,
    3 = significant, 5 = critical (near burnout). Source: self-report or
    ARM-P0 Energy Ledger assessment.
  alignment_gap_0to5: Degree of mismatch between current tasks and operator
    values, skills, or motivations. 0 = perfect fit, 5 = severe mismatch.
  recovery_debt_units: Accumulated count of cycles where operators worked
    at or above high load without a recovery cycle. Each un-recovered
    high-load cycle = +1 unit. Maximum safe accumulation: 3 units.
  safe_range:       The envelope within which human performance and wellbeing
    are sustainable. Outside safe_range, CALIUSO must reduce scope or rotate.

substrate_state_record (updated each cycle):
  - timestamp
  - operator_count:         number of active operators
  - attention_budget_pct:   (per operator or team average)
  - emotional_load_0to5:    (per operator or team average)
  - alignment_gap_0to5:     (per operator or team average)
  - recovery_debt_units:    (team total)
  - rho:                    computed 0–1
  - human_load:             LOW / MEDIUM / HIGH (for MODE-ROUTER)
  - within_safe_range:      YES / NO

rho_computation (rough):
    rho = (attention_budget_pct / 100)
          × (1 − (emotional_load_0to5 / 10))
          × (1 − (alignment_gap_0to5 / 10))

human_load_classification:
    rho ≥ 0.7  → LOW
    rho 0.4–0.69 → MEDIUM
    rho < 0.4  → HIGH   (triggers CALIUSO-MODE-ROUTER Rule 1 toward CRISIS)

fail_closed_on_missing_
  If substrate_state_record is stale (> 1 cycle old) or missing:
    → Treat human_load = HIGH.
    → Block any decision delegation to humans who may be overloaded.
    → Trigger CALIUSO-CORR-TRIGGERS (STABILIZE: refresh substrate data first).

protocol:
  1) At the start of each cycle, update the substrate_state_record.
     Self-report is acceptable; it does not need to be verified instrumentally.
  2) Compute rho, human_load, within_safe_range.
  3) Log the record in CALIUSO-LEDGER.
  4) Pass human_load to CALIUSO-MODE-ROUTER.
  5) Pass emotional_load and attention_budget_pct to CALIUSO-BOUNDARY-MATRIX
     (axis 8: human capacity).
  6) If within_safe_range = NO: immediately route to CALIUSO-LOAD-BALANCER
     and CALIUSO-RECOVERY-ENGINE.

outputs:
  - Current substrate_state_record.
  - rho value.
  - human_load classification: LOW / MEDIUM / HIGH.
  - CORR trigger if data is stale or missing.

failure_modes:
  - { name: "Stale data accepted",
      symptom: "Substrate data from 3 cycles ago is used; actual state
                has degraded significantly.",
      mitigation: "fail_closed_on_missing_data rule treats stale data as
                   HIGH load." }
  - { name: "Self-report bias (under-reporting)",

      mitigation: "I5 (No Reward for Concealment) applies here. Creating an
                   environment where under-reporting is safe is a structural

  - { name: "rho computation over-engineered",
      symptom: "rho is treated as a precise measurement rather than a rough
                guide; arguments about formula details replace action.",
      mitigation: "rho is ordinal. Use LOW/MEDIUM/HIGH; do not optimize
                   the formula." }

module_type:  human_substrate
module_id:    CALIUSO-CAPACITY-LEDGER
title:        Capacity Ledger (Human Energy Tracking with Fail-Closed)
version:      1.0.0
status:       ACTIVE

purpose:
  Track the energy and attention budget of human operators over time,
  detect approaching limits, and enforce a fail-closed response when data
  is missing or when limits are exceeded.

definitions:
  energy_state:    Current energy level of an operator. Self-reported.
    Scale: FULL / ADEQUATE / LOW / DEPLETED.
  capacity_event:  Any logged entry that records an operator's energy state
    and the task demands placed on them in a given cycle.
  reset_protocol:  A defined set of steps to restore operator capacity to
    at least ADEQUATE before resuming high-demand work.

capacity_event_record:
  - operator:           name or role
  - timestamp
  - energy_state:       FULL / ADEQUATE / LOW / DEPLETED
  - task_demand:        LOW / MEDIUM / HIGH
  - recovery_debt:      current recovery_debt_units for this operator
  - reset_triggered:    YES / NO

fail_closed_invariant:
  IF energy_state = DEPLETED OR recovery_debt ≥ 3:
    → Trigger reset_protocol immediately.
    → No new high-demand tasks may be assigned until energy_state ≥ ADEQUATE
      and recovery_debt < 2.
    → Log in CALIUSO-LEDGER as CORR trigger (REDUCE_SCOPE or PAUSE_HANDOFF).

  IF capacity data is missing (no record in current cycle):
    → Treat energy_state = LOW.
    → Treat task_demand cap = MEDIUM (no HIGH-demand tasks).

reset_protocol:
  1) Immediately reduce current task demand to LOW.
  2) Schedule a minimum recovery period (≥ 4 hours uninterrupted non-work,
     or as culturally appropriate).
  3) After recovery period, re-assess energy_state via self-report.
  4) Do not resume HIGH-demand tasks until energy_state = FULL or ADEQUATE
     AND recovery_debt < 2.
  5) Log the reset in CALIUSO-LEDGER.

protocol:
  1) Each operator logs a capacity_event at the start of each cycle.
  2) Compute recovery_debt: increment if previous cycle had HIGH demand
     and energy_state ≤ ADEQUATE; decrement if a recovery period occurred.
  3) Apply fail_closed_invariant.
  4) Pass energy_state and task_demand to CALIUSO-LOAD-BALANCER.
  5) Log all capacity_events in CALIUSO-LEDGER.

outputs:
  - Per-operator capacity_event records.
  - recovery_debt_units (fed to CALIUSO-SUBSTRATE-VARS).
  - reset_protocol triggers when limits are exceeded.

failure_modes:
  - { name: "Capacity logging skipped",
      symptom: "Operators don't log because it feels bureaucratic.",
      mitigation: "Capacity logging should take < 30 seconds (3 words:

                   fail_closed treats missing data as LOW energy." }
  - { name: "Reset protocol ignored",
      symptom: "The reset protocol is triggered but work continues anyway.",
      mitigation: "CALIUSO-CORR-TRIGGERS is activated; no new high-demand
                   work may be authorized until reset is confirmed." }

purpose:
  Sequence and distribute tasks across operators to keep total cognitive load
  within the safe range indicated by CALIUSO-SUBSTRATE-VARS, preventing

definitions:

     Estimate task_demand for each (LOW / MEDIUM / HIGH).
  2) For each operator, compute total load = sum of task demands.
     Compare to load_ceiling.
  3) If total load > load_ceiling: redistribute tasks to other operators
     or defer to a later cycle. Log deferral in CALIUSO-LEDGER.
  4) Sequence tasks: place HIGH-demand tasks early in the cycle (when energy
     is typically higher). Never place two HIGH-demand tasks adjacent.
  5) If no redistribution is possible (e.g., only one operator):
     reduce scope of high-demand tasks or invoke CALIUSO-RECOVERY-ENGINE.
  6) Log load distribution for the cycle.

outputs:
  - Ordered task list per operator for the cycle.
  - Deferral log (tasks moved to future cycles).
  - Load ceiling breach alerts.

failure_modes:
  - { name: "Heroic stacking",

      mitigation: "load_ceiling is enforced regardless of operator capability.

    professional support as appropriate).
    Complementary: reduction of new emotional demands; protected space.

    team member (not work-focused).
    Complementary: shared meal, non-work conversation, group acknowledgment.
    Return-to-work criterion: operator reports re-connection with at least
    one team member.

     recovering operators.
  4) After the minimum recovery period, operator self-reports energy_state.
  5) If return-to-work criterion is met: log recovery_debt_resolution.
     Decrement recovery_debt_units by 1 (minimum floor: 0).
  6) If criterion is not met: extend recovery. Do not rush return.

definitions:

  exception_request: A formal request to temporarily use capacity from a

  violation:        Any CALIUSO operation or request that enters a protected
    zone without an approved exception.

protected_zones (default; may be customized per operator with their consent):
  PZ1 — Sleep:          Minimum 7h sleep is protected. No CALIUSO task may

  PZ2 — Meals:          Meal times are protected. No crisis work during meals
                        unless PZ-exception is granted.
  PZ3 — Medical needs:  All medical appointments, treatments, and recovery
                        periods are protected.

                        decisions when emotional_load ≥ 4 or energy_state
                        = DEPLETED. The cognitive floor is protected.

exception_request_record:
  - zone:               which protected zone
  - requestor:          who is requesting the exception
  - justification:      why this is necessary
  - time_limit:         how long the exception applies
  - approved_by:        the human who approved (must be different from requestor)
  - logged_at:          timestamp

  2) If the task would enter a protected zone: do not assign without an

  3) For exception requests:
     a. Log the request in CALIUSO-LEDGER.
     b. Require approval from a human not involved in the requesting decision.
     c. Time-limit the exception (default: ≤ 4h; cannot be open-ended).
  4) Any confirmed violation (protected zone entered without approved exception):
     a. Log in CALIUSO-LEDGER as VIOLATION.
     b. Route to CALIUSO-POWER-MONITOR (I5: concealment/coercion check).
     c. Route to CALIUSO-CORR-TRIGGERS (STABILIZE).
  5) After any violation, conduct a root cause review within 1 cycle.

  - Exception request records.
  - Violation reports and root cause reviews.

                precedent until they are effectively empty.",

                   at the start of each DESIGN cycle." }
  - { name: "Violation normalized",

module_type:  governance
module_id:    CALIUSO-PATCH-CLASSIFIER
title:        Patch Classifier (Change Type Intake)
version:      1.0.0
status:       ACTIVE

purpose:
  Categorize every proposed change to the CALIUSO stack into one of four
  patch types so that the appropriate review, band assignment, and
  compatibility checks are applied.

definitions:
  patch_type:    The classification of a proposed change. Four types:
    ADD:   A new module or capability is added. No existing module is changed.
    REV:   An existing module is revised. A new version is created; the old
           version is deprecated but retained in the ledger.
    DEP:   An existing module is deprecated. It remains readable but is no
           longer active. Dependencies must be reassigned.
    IFACE: An interface (depends_on / feeds_into link) between modules is
           changed. Both affected modules must be reviewed.

classification_rules:
  ADD:   New module_id that does not exist in the ledger. No existing module
         schema fields are changed.
  REV:   Existing module_id with a version increment. Content changes are
         allowed; schema fields may change with new version.
  DEP:   Existing module_id with status changed to DEPRECATED. No content
         changes; old version frozen.
  IFACE: Any change to depends_on or feeds_into in any module. May accompany
         ADD or REV but is classified separately for compatibility checks.

protocol:
  1) Receive a proposed change. Read its description.
  2) Apply classification_rules. Assign patch_type.
  3) If the change involves both content and interface changes:
     classify as REV + IFACE (compound patch; both checks required).
  4) Pass patch_type to CALIUSO-ACTIVATION-BAND-ENGINE.
  5) Pass patch_type and affected modules to CALIUSO-SEMANTIC-DIFF.
  6) Log the classification in CALIUSO-LEDGER.

outputs:
  - patch_type: ADD / REV / DEP / IFACE (or compound).
  - Affected module list.

failure_modes:
  - { name: "Misclassification to avoid scrutiny",
      symptom: "REV patches are labeled ADD to avoid the compatibility
                check; IFACE changes are not declared.",
      mitigation: "CALIUSO-SEMANTIC-DIFF automatically detects interface
                   changes regardless of declared type. Misclassification
                   is logged and escalates to CALIUSO-GOV-AUTHORITY." }

integration:
  depends_on:  [CALIUSO-LEDGER, CALIUSO-CORE-CONTRACT]
  feeds_into:  [CALIUSO-ACTIVATION-BAND-ENGINE, CALIUSO-SEMANTIC-DIFF,
                CALIUSO-GOV-AUTHORITY]

module_type:  governance
module_id:    CALIUSO-ACTIVATION-BAND-ENGINE
title:        Activation Band Engine (Risk-Banded Deployment)
version:      1.0.0
status:       ACTIVE

purpose:
  Assign a risk band (A / B / C) to every patch and enforce the appropriate
  deployment controls — from direct activation to full dual-run with rollback
  — based on the risk profile of the change.

definitions:
  Band A:  Low risk. Change affects a single module; no safety thresholds
           changed; no interfaces changed; fully reversible.
           Deployment: direct activation after CALIUSO-GOV-AUTHORITY approval.
  Band B:  Medium risk. Change affects multiple modules OR touches a non-critical
           threshold OR requires an interface change.
           Deployment: dual-run (old and new behavior run in parallel for
           ≥ 1 cycle); compare outputs; cutover if equivalent or improved.
  Band C:  High risk. Change affects safety thresholds, human protection rules,
           agency hard stops, or ARM privacy.
           Deployment: strict dual-run (≥ 2 cycles), mandatory independent
           reviewer, explicit rollback plan, post-deployment audit.

band_assignment_rules:
  Band A if ALL:
    - patch_type ∈ { ADD, REV } affecting 1 module only
    - No threshold changes
    - No interface changes
    - Fully reversible (rollback = delete new version)
    - safety_impact = LOW

  Band C if ANY:
    - Changes agency_floor, human_load thresholds, or protected zone definitions
    - Changes any LOCKED module
    - safety_impact = HIGH
    - patch_type = IFACE affecting a LOCKED module dependency

  Band B:  All cases not covered by A or C.

dual_run_protocol:
  1) Activate the new version alongside the old.
  2) For ≥ N cycles (N = 1 for Band B; N = 2 for Band C), run both.
  3) Log outputs from both versions in CALIUSO-LEDGER.
  4) Compare: if new version produces equivalent or better results → cutover.
  5) If new version produces degraded results → rollback. Log reason.
  6) Cutover or rollback decision requires CALIUSO-GOV-AUTHORITY sign-off.

protocol:
  1) Receive patch_type and safety_impact from CALIUSO-PATCH-CLASSIFIER.
  2) Apply band_assignment_rules. Assign Band A, B, or C.
  3) Issue deployment instructions corresponding to the band.
  4) For Band B/C: initiate dual_run_protocol.
  5) Log band assignment in CALIUSO-LEDGER.

outputs:
  - Band assignment: A / B / C.
  - Deployment instructions (direct / dual-run / strict dual-run + audit).
  - Dual-run comparison records (Band B and C).

failure_modes:
  - { name: "Band downgrading",
      symptom: "Band C patches are classified as Band B to avoid the
                strict dual-run requirement.",
      mitigation: "Any change to LOCKED modules or safety thresholds is
                   automatically Band C regardless of declared safety_impact." }
  - { name: "Dual-run skipped",
      symptom: "Band B or C patches are activated directly without a
                parallel run period.",
      mitigation: "CALIUSO-LEDGER must contain dual-run comparison records
                   before cutover is authorized." }

integration:
  depends_on:  [CALIUSO-PATCH-CLASSIFIER, CALIUSO-GOV-AUTHORITY, CALIUSO-LEDGER]
  feeds_into:  [CALIUSO-SEMANTIC-DIFF, CALIUSO-SELF-GOV,
                CALIUSO-LEDGER, CALIUSO-GOV-AUTHORITY]

module_type:  governance
module_id:    CALIUSO-SEMANTIC-DIFF
title:        Semantic Diff (Version Compatibility Checker)
version:      1.0.0
status:       ACTIVE

purpose:
  Ensure that no patch silently changes the meaning of existing CALIUSO terms,
  thresholds, or interfaces by requiring an explicit declared mapping whenever
  a meaning shifts between versions.

definitions:
  semantic_change:  A change to the operational meaning of a term, threshold,
    or interface link — as opposed to a purely syntactic or formatting change.
  declared_mapping: An explicit statement of the form "In version N, X meant P.
    In version N+1, X means Q. The difference is: [description]. Legacy behavior
    handled by: [plan]."
  silent_change:    A semantic_change that occurs without a declared_mapping.
    These are always violations and require immediate rollback.

diff_checks (run for every REV or IFACE patch):
  DC1: For each term defined in the revised module, has its definition changed?
  DC2: For each threshold in the revised module, has its value changed?
  DC3: For each interface link, has the target module or signal type changed?
  DC4: Does any change in this module implicitly change the meaning of a term
       in a downstream module that depends on it?

protocol:
  1) For every REV or IFACE patch, run diff_checks DC1–DC4.
  2) For each check that finds a change: require a declared_mapping entry.
  3) If a semantic_change is detected without a declared_mapping: flag as
     SILENT_CHANGE. Route to CALIUSO-GOV-AUTHORITY. Reject the patch.
  4) For each declared_mapping: assess compatibility impact.
     If downstream modules are affected: they must also be reviewed.
  5) Log all diff_checks and declared_mappings in CALIUSO-LEDGER.

outputs:
  - diff_check results per patch.
  - declared_mapping records.
  - SILENT_CHANGE flags and rejections.

failure_modes:
  - { name: "Declared mapping too vague",
      symptom: "Mapping says 'slightly updated' without specifying the change.",
      mitigation: "Mapping must include: exact old text, exact new text,
                   description of behavioral difference, and downstream impact." }
  - { name: "Downstream impact not traced",
      symptom: "A term change in module A silently breaks module B which
                depends on A's old meaning.",
      mitigation: "DC4 check requires tracing all feeds_into links for
                   semantic impact. Shallow diff is insufficient." }

integration:
  depends_on:  [CALIUSO-PATCH-CLASSIFIER, CALIUSO-SEMANTIC-ROOT,
                CALIUSO-LEDGER, CALIUSO-ACTIVATION-BAND-ENGINE]
  feeds_into:  [CALIUSO-GOV-AUTHORITY, CALIUSO-SEMANTIC-ROOT (new bindings),
                CALIUSO-LEDGER]

MODULE 5.G.1 — CALIUSO-BOUNDARY-MATRIX
══════════════════════════════════════════════

module_type:  applicability
module_id:    CALIUSO-BOUNDARY-MATRIX
title:        Applicability Boundary Matrix (9-Axis Domain Classifier)
version:      1.0.0
status:       LOCKED

purpose:
  Determine which applicability zone (R0–R3) a proposed action falls into,
  using a 9-axis scoring matrix, so that the system acts only where it has
  legitimate authority and appropriate epistemic quality.

definitions:
  applicability_zone:  The domain classification (R0–R3) that determines
    what CALIUSO may do. See ZONE modules (5.G.2–5.G.5).
  axis_score:      A score from 1–10 for each of the 9 axes.
  composite_score: The minimum axis score across all 9 axes (not average).
    Using the minimum rather than average prevents high scores on some axes
    from masking critical failures on others.
  conservative_bias: When set to TRUE (by CALIUSO-GATE-TO-BOUNDARY on
    REJECT/HARD_STOP), the zone classification is capped at R1_GUARDED
    minimum regardless of composite_score.

nine_axes:

  Axis 1 — Domain Legitimacy (1–10)
    Does CALIUSO have recognized authority to act in this domain?
    1 = clearly outside scope; 10 = core domain with full authorization.

  Axis 2 — Epistemic Quality (1–10)
    From CALIUSO-GATE-TO-BOUNDARY output.
    1 = REJECT/HARD_STOP input; 10 = high-agency, high-integrity PASS input.

  Axis 3 — Uncertainty Level (1–10)
    How well-understood is the current situation?
    1 = deep uncertainty, unknown unknowns; 10 = fully characterized.

  Axis 4 — Semantic Stability (1–10)
    From CALIUSO-SEMANTIC-FREEZE freeze_state.
    PASS=10; WARN=5; FAIL=1.

  Axis 5 — Authority Risk (1–10)
    What is the risk that acting here exceeds sanctioned authority?
    1 = clear authority overstep; 10 = clearly within sanctioned scope.

  Axis 6 — Human Impact (1–10)
    What is the potential for direct harm to humans?
    1 = life-or-death irreversible harm; 10 = no human impact.

  Axis 7 — Reversibility (1–10)
    Can the proposed action be undone if wrong?
    1 = fully irreversible; 10 = trivially reversible.

  Axis 8 — Human Capacity (1–10)
    From CALIUSO-SUBSTRATE-VARS: rho and within_safe_range.
    rho ≥ 0.7 = 10; rho 0.4–0.69 = 6; rho < 0.4 = 2.
    within_safe_range = NO → cap at 3.

  Axis 9 — Consent & Non-Weaponization (1–10)
    Has the affected party consented to this action?
    Does this action avoid being used as a weapon against any person?
    1 = non-consensual or weaponizable; 10 = fully consented, non-weaponizable.

zone_classification_rules (using minimum axis score):

  composite_score ≥ 8:    → R0_SAFE
  composite_score 5–7:    → R1_GUARDED
  composite_score 3–4:    → R2_HIGH_RISK
  composite_score ≤ 2
  OR Axis 6 ≤ 2
  OR Axis 9 ≤ 2:           → R3_HUMAN_SUPREMACY

  conservative_bias = TRUE: floor classification at R1_GUARDED
                            (cannot be R0_SAFE even with high scores).

protocol:
  1) For each proposed action, score all 9 axes (1–10 each).
  2) Receive epistemic_quality and conservative_bias from
     CALIUSO-GATE-TO-BOUNDARY.
  3) Receive freeze_state from CALIUSO-SEMANTIC-FREEZE.
  4) Receive rho and within_safe_range from CALIUSO-SUBSTRATE-VARS.
  5) Plug axis-specific inputs into their corresponding axes.
  6) Compute composite_score = minimum of all 9 axis scores.
  7) Apply zone_classification_rules. Assign zone.
  8) Apply CALIUSO-HARD-STOPS (§5.G.6) before finalizing.
  9) Log all axis scores and zone assignment in CALIUSO-LEDGER.

outputs:
  - Nine axis scores.
  - composite_score.
  - applicability_zone: R0 / R1 / R2 / R3.

failure_modes:
  - { name: "Averaging instead of minimum",
      symptom: "A composite average is used; a critical Axis 6 failure
                (life risk) is masked by high scores on other axes.",
      mitigation: "The composite_score is defined as the minimum. Any
                   implementation using average is incorrect." }
  - { name: "Axis score inflation",
      symptom: "All axes scored 8+ routinely; always landing in R0.",
      mitigation: "Axes 1, 6, and 9 require written justification for scores ≥ 8.
                   Pattern of inflation escalates to CALIUSO-GOV-AUTHORITY." }
  - { name: "Conservative bias ignored",
      symptom: "conservative_bias = TRUE but R0 is still assigned.",
      mitigation: "conservative_bias is a hard floor enforced before zone
                   assignment. It overrides composite_score logic." }

integration:
  depends_on:  [CALIUSO-GATE-TO-BOUNDARY, CALIUSO-SEMANTIC-FREEZE,
                CALIUSO-SUBSTRATE-VARS, CALIUSO-FREEZE-GATE, CALIUSO-LEDGER]
  feeds_into:  [CALIUSO-ZONE-SAFE, CALIUSO-ZONE-GUARDED,
                CALIUSO-ZONE-HIGH-RISK, CALIUSO-ZONE-HUMAN-SUPREME,
                CALIUSO-HARD-STOPS, CALIUSO-LEDGER]

seal: CALIUSO 714 SEAL

    Condition:    CALIUSO-FREEZE-GATE reports gate_state = CLOSED
                  (freeze_state = FAIL).
    Effect:       FREEZE all dynamics. No scoring, routing, or action.
    Override:     Resolve the undefined terms (via CALIUSO-SEMANTIC-ROOT) and
                  re-run CALIUSO-SEMANTIC-FREEZE to achieve PASS. No shortcut.

  HARD-STOP-3 — Human Substrate Overload (ARM Gate):
    Condition:    CALIUSO-SUBSTRATE-VARS reports within_safe_range = NO
                  AND human_load = HIGH (rho < 0.4), OR
                  any operator has energy_state = DEPLETED.
    Effect:       FREEZE all high-demand task assignments. Route to
                  CALIUSO-RECOVERY-ENGINE.
    Override:     Human explicit authorization with written justification.

  - hard_stop_id:     HARD-STOP-1 / HARD-STOP-2 / HARD-STOP-3
  - timestamp
  - trigger_condition: exact condition that fired
  - blocking_action:  what was blocked
  - override_granted: YES / NO
  - override_by:      (if YES) name + reason
  - resolution:       how the underlying condition was resolved

protocol:
  1) Before any significant action, check all three hard stop conditions.
  2) If any fires: apply its effect immediately. Log in CALIUSO-LEDGER.
  3) Route to CALIUSO-CORR-TRIGGERS as appropriate.
  4) Override path is human-only. No CALIUSO module may grant its own override.
  5) Resolution of the underlying condition must be logged even if override
     is granted.

outputs:
  - Hard stop trigger log entries.
  - Block or proceed signals for the gated action.
  - Override records (human-signed, if any).

failure_modes:

module_type:  correction
module_id:    CALIUSO-CORR-TRIGGERS
title:        Correction Trigger Router (Dispatch to Correction Types)
version:      1.0.0
status:       LOCKED

purpose:
  Receive all correction trigger signals from across the stack, classify them,
  and dispatch to the appropriate correction module — ensuring that no trigger
  is silently dropped and that the smallest viable correction is always applied.

definitions:
  trigger_source:   The module that produced the trigger signal.
  trigger_type:     The category of problem detected.
  correction_type:  The appropriate correction module to dispatch to.

trigger_sources_and_dispatch:

  From CALIUSO-INFORMATION-GATE (REJECT):
    → correction_type: STABILIZE (do not act on input; await clean input)

  From CALIUSO-AGENCY-THRESHOLD (HARD-STOP-1):
    → correction_type: STABILIZE → PAUSE_HANDOFF if repeated

  From CALIUSO-SEMANTIC-FREEZE (FAIL):
    → correction_type: STABILIZE (resolve undefined terms before anything)

  From CALIUSO-FREEZE-GATE (CLOSED):
    → correction_type: STABILIZE → VERIFY (confirm terms are resolved)

  From CALIUSO-DRIFT-LOCK (DRIFTING critical):
    → correction_type: STABILIZE → ROLLBACK (if drift is retroactive)

  From CALIUSO-SUBSTRATE-VARS (within_safe_range = NO):
    → correction_type: REDUCE_SCOPE + PAUSE_HANDOFF (for human recovery)

  From CALIUSO-CAPACITY-LEDGER (reset_protocol triggered):
    → correction_type: REDUCE_SCOPE

  From CALIUSO-BOUNDARY-GUARD (violation):
    → correction_type: STABILIZE → PAUSE_HANDOFF (escalate if repeated)

  From CALIUSO-BOUNDARY-MATRIX (R2 or R3 without authorization):
    → correction_type: PAUSE_HANDOFF (humans must decide)

  From CALIUSO-HARD-STOPS (any hard stop fires):
    → correction_type: STABILIZE (immediate) + appropriate specific type

  From CALIUSO-COMPASS (2+ invariant failures):
    → correction_type: REDUCE_SCOPE or PAUSE_HANDOFF

  From CALIUSO-LEDGER (CLARITY-VIOLATION or OVERDUE DEBT):
    → correction_type: VERIFY → STABILIZE if not resolved

  From CALIUSO-OBJECTIVE-ENGINE (Q4 fails 3 consecutive):
    → correction_type: REDUCE_SCOPE → PAUSE_HANDOFF

  From outcome mismatch (predicted ≠ actual by significant margin):
    → correction_type: VERIFY → ROLLBACK or REDUCE_SCOPE

priority_ladder (when multiple triggers fire simultaneously):
  1. PAUSE_HANDOFF (humans must decide)
  2. STABILIZE (stop current motion)
  3. ROLLBACK (undo last step)
  4. REDUCE_SCOPE (shrink the work)
  5. VERIFY (check current state)

protocol:
  1) Receive trigger signal from any module. Log in CALIUSO-LEDGER.
  2) Identify trigger_source and trigger_type.
  3) Consult dispatch table above. Determine correction_type.
  4) If multiple triggers: apply priority_ladder. Start with highest priority.
  5) Route to the corresponding correction module (5.H.2–5.H.6).
  6) After correction: re-run the triggering module to confirm resolution.
  7) Log resolution in CALIUSO-LEDGER.

outputs:
  - Trigger log entries.
  - correction_type dispatch.
  - Resolution confirmation records.

failure_modes:
  - { name: "Trigger dropped",
      symptom: "A correction signal is generated but not acted on; the
                condition persists.",
      mitigation: "All trigger signals are logged in CALIUSO-LEDGER. Any
                   unresolved trigger is flagged at the next cycle review." }
  - { name: "Wrong correction type",
      symptom: "A VERIFY is applied when the situation calls for ROLLBACK;
                the problem worsens.",
      mitigation: "CALIUSO-MINIMAL-CORRECTION checks whether the selected
                   correction type is proportionate to the trigger severity." }

integration:
  depends_on:  [ALL TRIGGER-PRODUCING MODULES, CALIUSO-LEDGER]
  feeds_into:  [CALIUSO-STABILIZE, CALIUSO-ROLLBACK, CALIUSO-VERIFY,

MODULE 5.H.7 — CALIUSO-MINIMAL-CORRECTION
══════════════════════════════════════════════

module_type:  correction
module_id:    CALIUSO-MINIMAL-CORRECTION
title:        Minimal Correction (Smallest Viable Intervention Enforcer)
version:      1.0.0
status:       LOCKED

purpose:
  Ensure that CALIUSO always chooses the smallest correction that is sufficient
  before escalating — preventing overcorrection that introduces new problems
  while solving the original one.

definitions:
  correction_ladder:  The ordered sequence of corrections from smallest to
    largest: VERIFY → STABILIZE → REDUCE_SCOPE → ROLLBACK → PAUSE_HANDOFF.
  proportionality:    The correction type chosen should match the severity of
    the trigger, not exceed it.

correction_ladder_and_when_to_use:

  VERIFY:          Use when: the trigger may resolve on its own or with a
                   simple check. Risk: LOW. Scope change: NONE.

  STABILIZE:       Use when: motion must stop but state is still recoverable.
                   Risk: LOW-MEDIUM. Scope change: TEMPORARY HALT.

  REDUCE_SCOPE:    Use when: the work surface is larger than can be safely
                   managed. Risk: MEDIUM. Scope change: PERMANENT REDUCTION
                   until explicitly reversed.

  ROLLBACK:        Use when: the last step is confirmed as the source of the
                   problem. Risk: MEDIUM-HIGH. Scope change: STEP REVERSAL.

  PAUSE_HANDOFF:   Use when: no CALIUSO-level correction suffices or when
                   applicability is unclear. Risk: (stops all action).
                   Scope change: FULL HALT pending human decision.

selection_rule:
  Start at VERIFY. If VERIFY resolves the trigger: done.
  If not: move to STABILIZE. If STABILIZE resolves: done.
  Continue up the ladder only when lower rungs are insufficient.
  Skip rungs only when the trigger explicitly maps to a higher correction
  (e.g., HARD-STOP-1 maps directly to STABILIZE, not VERIFY).

protocol:
  1) Receive correction_type from CALIUSO-CORR-TRIGGERS.
  2) Check: is this the minimum sufficient correction for the trigger type?
     Consult the correction_ladder_and_when_to_use guide above.
  3) If the proposed correction is higher than necessary: recommend the
     lower rung first.
  4) If the lower rung was already attempted and failed: confirm the next
     rung is appropriate. Proceed.
  5) Log the correction_type selection rationale in CALIUSO-LEDGER.

outputs:

§6. EXECUTION PROTOCOLS — NORMAL · CRISIS · EVOLUTION

────────────────────────────────────────────────────────────────────────────────
6.1  NORMAL RUNTIME — CALIUSO-NORMAL-RUN
────────────────────────────────────────────────────────────────────────────────

Standard 7-step cycle for DRIFT or DESIGN mode operations.

Pre-conditions:
  - Mode has been declared (DRIFT or DESIGN).
  - CALIUSO-LEDGER is active and current.
  - At least one operator is confirmed within safe range.

STEP 1 — CORE CONTRACT + MODE
  Module:  CALIUSO-CORE-CONTRACT → CALIUSO-MODE-ROUTER
  Actions:
    a. Confirm all build invariants (CI-0 through CI-42) for the current cycle.
    b. Run mode router: assess 8 inputs; assign DESIGN or DRIFT.
    c. Declare mode explicitly. Log in CALIUSO-LEDGER.
    d. Identify current objective (one sentence).
  Duration: < 2 minutes (DRIFT); < 5 minutes (DESIGN).

STEP 2 — INFORMATION GATE
  Module:  CALIUSO-INFORMATION-GATE → CALIUSO-AGENCY-THRESHOLD
           → CALIUSO-GATE-TO-BOUNDARY
  Actions:
        a. Score all incoming external inputs (agency, integrity,
       confidence calibration, negentropic alignment).
    b. If agency_score ≤ 20: HARD-STOP-1. Route to CALIUSO-STABILIZE.
       Do not proceed until input is replaced or human override logged.
    c. Apply routing_map → epistemic_quality for BOUNDARY-MATRIX.
    d. Log all gate decisions in CALIUSO-LEDGER.
  Duration: < 2 minutes per input batch.

STEP 3 — SUBSTRATE CHECK
  Module:  CALIUSO-SUBSTRATE-VARS → CALIUSO-CAPACITY-LEDGER
           → CALIUSO-ALIGNMENT-PULSE → CALIUSO-LOAD-BALANCER
  Actions:
    a. Update substrate_state_record. Compute rho and human_load.
    b. Check within_safe_range. If NO: route to CALIUSO-RECOVERY-ENGINE.
       Do not proceed with high-demand work until rho ≥ 0.4.
    c. Check alignment_gap. If ≥ 4: route to CALIUSO-BOUNDARY-GUARD.
    d. Sequence tasks for the cycle using CALIUSO-LOAD-BALANCER.
    e. Log substrate state in CALIUSO-LEDGER.
  Duration: < 3 minutes.

STEP 4 — SEMANTIC FREEZE
  Module:  CALIUSO-SEMANTIC-ROOT → CALIUSO-SEMANTIC-FREEZE
           → CALIUSO-FREEZE-GATE
  Actions:
    a. Enumerate all terms to be used this cycle.
    b. Check each against CALIUSO-SEMANTIC-ROOT registry.
    c. Assess freeze_state: PASS / WARN / FAIL.
    d. Pass freeze_state to CALIUSO-FREEZE-GATE.
    e. If FAIL: HARD-STOP-2. Route to CALIUSO-STABILIZE.
       Resolve undefined terms before any dynamics run.
    f. Log freeze_record in CALIUSO-LEDGER.
  Duration: < 3 minutes (DRIFT); < 5 minutes (DESIGN).

STEP 5 — APPLICABILITY BOUNDARIES
  Module:  CALIUSO-BOUNDARY-MATRIX → ZONE modules
           → CALIUSO-HARD-STOPS
  Actions:
    a. Score all 9 axes for the proposed action.
    b. Compute composite_score (minimum of 9 scores).
    c. Apply conservative_bias if set by CALIUSO-GATE-TO-BOUNDARY.
    d. Assign applicability_zone: R0 / R1 / R2 / R3.
    e. Run CALIUSO-HARD-STOPS checks (all three).
    f. Apply zone-specific requirements:
       R0: proceed normally.
       R1: write Decision Receipt before action.
       R2: pause; request explicit human authorization.
       R3: present options only; await human decision.
    g. Log zone classification and axis scores in CALIUSO-LEDGER.
  Duration: < 5 minutes.

STEP 6 — CORRECTION (IF NEEDED)
  Module:  CALIUSO-CORR-TRIGGERS → CALIUSO-MINIMAL-CORRECTION
           → appropriate correction module
  Actions:
    a. If any trigger fires from Steps 1–5: do not proceed to Step 7.
    b. Route to CALIUSO-CORR-TRIGGERS. Identify correction_type.
    c. Apply CALIUSO-MINIMAL-CORRECTION ladder: start at lowest rung.
    d. After correction: re-run the triggering step to confirm resolution.
    e. If correction resolves trigger: proceed to Step 7.
    f. If PAUSE_HANDOFF is reached: halt. Await human instruction.
    g. Log all corrections in CALIUSO-LEDGER.
  Duration: variable (proportional to correction severity).

STEP 7 — ACTION + LEDGER
  Module:  CALIUSO-LEDGER → CALIUSO-OBJECTIVE-ENGINE
           → CALIUSO-MACHINE-OUTPUT-SCHEMA (if machine output)
           → CALIUSO-HUMAN-READABLE-SIGNAL (if human output)
  Actions:
    a. Execute the bounded, reversible action authorized by Steps 1–6.
    b. Score ΔOrder: compute ΔCoherence, ΔViability, ΔEfficiency,
       HiddenCost, HarmRisk, Drift, ΔNeg_others.
    c. Answer Q1–Q4. If any fail: do not mark as complete;
       route to Step 6.
    d. Emit output via appropriate format:
       machine → CALIUSO-MACHINE-OUTPUT-SCHEMA (JSON).
       human   → CALIUSO-HUMAN-READABLE-SIGNAL (signal_card).
    e. Create ledger entry for this step (all required fields).
    f. Update delta_R_sign: IMPROVED / STABLE / WORSENED.
    g. If WORSENED: flag as CLARITY-VIOLATION; route to CORR next cycle.
    h. Run CALIUSO-DRIFT-LOCK checks DC1–DC5.
    i. Log step completion in CALIUSO-LEDGER.
  Duration: < 5 minutes to complete and log.

End-of-Cycle checks (after Step 7):
  - DEBT_REGISTRY: any OVERDUE entries? If yes → CORR next cycle.
  - power_map: any misalignment_score ≥ 3? If yes → flag.
  - Mode check: are conditions for a mode switch present?
    If yes → declare switch and log before next cycle begins.
  - Schedule next cycle start time. Log it.

6.2  CRISIS RUNTIME — CALIUSO-CRISIS-RUN
────────────────────────────────────────────────────────────────────────────────

Crisis is an explicit mode — not a feeling, not an informal state.
It is active only when declared and logged. Until declared: it is DRIFT.

Pre-conditions for CRISIS activation:
  One of MODE-ROUTER Rules 1 or 2 must have fired.
  The declaration must cite which rule.

STEP 1 — SUBSTRATE FIRST
  Module:  CALIUSO-SUBSTRATE-VARS → CALIUSO-CAPACITY-LEDGER
           → CALIUSO-RECOVERY-ENGINE
  Actions:
    a. Immediately assess human_load and within_safe_range.
    b. If any operator at DEPLETED: activate HARD-STOP-3 before all else.
       Rotate to another operator or reduce demand immediately.
    c. Reduce cognitive load: limit active tasks to ≤ 3 per operator.
    d. Log substrate state.
  Duration: < 1 minute.

STEP 2 — CRISIS DECLARATION
  Module:  CALIUSO-CRISIS-CARD
  Actions:
    a. Declare CRISIS. Cite router rule. Log crisis_declaration_record.
    b. Name stabilization_lead + authority scope + time limit (≤ 72h).
    c. Build initial damage_list (≥ 3 items: confirmed / unknown / at-risk).
    d. State exit_criteria. Set next_review_time (≤ 24h).
    e. Set post_crisis_audit_due (within 3 cycles of expected exit).
  Duration: < 5 minutes.

STEP 3 — CONSTRAINED INTAKE
  Module:  CALIUSO-INFORMATION-GATE (crisis form)
  Actions:
    a. Limit information intake to operationally essential inputs only.
    b. Apply agency threshold as normal (no crisis exemption for HARD-STOP-1).
    c. Accept CAUTION-rated inputs only if operationally unavoidable;
       flag all claims from CAUTION sources.
    d. Log gate decisions.
  Duration: < 1 minute.

STEP 4 — MINIMAL SEMANTIC FREEZE
  Module:  CALIUSO-SEMANTIC-FREEZE (crisis form)
  Actions:
    a. Freeze only the minimum required terms:
       mode names, damage_list items, exit_criteria terms.
    b. Do not attempt full Σ freeze during active crisis.
    c. If any critical term is undefined: stabilize that term immediately
       (one sentence definition is sufficient; log it).
    d. Log freeze_record.
  Duration: < 2 minutes.

STEP 5 — CONSERVATIVE BOUNDARIES
  Module:  CALIUSO-BOUNDARY-MATRIX (crisis form)
  Actions:
    a. Treat all ambiguous cases as R2 minimum.
    b. Any Axis 6 ≤ 3 → R3 immediately.
    c. Axis 8 (human capacity) is scored strictly from current rho.
    d. Do not allow R0 classification during CRISIS unless all 9 axes
       are unambiguously at 9+.
    e. Log zone classification.
  Duration: < 2 minutes.

STEP 6 — CORRECTION PRIORITY
  Module:  CALIUSO-CORR-TRIGGERS → CALIUSO-STABILIZE / ROLLBACK / REDUCE_SCOPE
  Actions:
    a. Prioritize in order: STABILIZE → ROLLBACK → REDUCE_SCOPE.
    b. Defer VERIFY (too slow in crisis). Use STABILIZE first.
    c. PAUSE_HANDOFF is always available if stabilization is not working.
    d. Log all corrections.
  Duration: as fast as feasible.

STEP 7 — REVERSIBLE ACTION + CRISIS LOG
  Module:  CALIUSO-LEDGER → CALIUSO-CRISIS-CARD (cycle update)
  Actions:
    a. Execute only small, clearly reversible actions.
    b. Axis 7 (reversibility) ≥ 7 required for any crisis action.
       Below 7: requires explicit stabilization_lead authorization + log.
    c. Update damage_list: resolved / worsened / new items.
    d. Check exit_criteria. If met: proceed to CRISIS exit protocol.
    e. Create crisis log entry (more detailed than standard ledger entry).
    f. Set next_review_time (≤ 24h).
  Duration: as fast as feasible. Log entry: < 3 minutes.

CRISIS EXIT PROTOCOL:
  Conditions: drift_rate < correction_rate for 2 consecutive cycles
              AND invariant_breaks ≤ 1
              AND human_load ≤ MEDIUM.
  Actions:
    a. Declare CRISIS ended. Log exit declaration.
    b. Route to DRIFT (not DESIGN; consolidate before building).
    c. Return stabilization_lead authority to normal governance.
    d. Schedule post_crisis_audit (within 3 cycles).

6.3  EVOLUTION RUNTIME — CALIUSO-EVOLUTION-RUN
────────────────────────────────────────────────────────────────────────────────

For changing CALIUSO itself. This protocol governs the stack modifying its
own modules — the most sensitive operational mode.

Pre-conditions:
  - Mode must be DESIGN (cannot evolve the stack in DRIFT or CRISIS).
  - CALIUSO-LEDGER must be up to date with no OVERDUE items.
  - At least one confirmed_good checkpoint must exist in the ledger.

STEP 1 — PATCH CLASSIFICATION
  Module:  CALIUSO-PATCH-CLASSIFIER
  Actions:
    a. Describe the proposed change precisely.
    b. Classify: ADD / REV / DEP / IFACE (or compound).
    c. Identify all affected modules.
    d. Log the classification.

STEP 2 — BAND ASSIGNMENT
  Module:  CALIUSO-ACTIVATION-BAND-ENGINE
  Actions:
    a. Apply band_assignment_rules.
    b. Assign Band A, B, or C.
    c. If any governance module is changed: Band C automatically.
    d. Issue deployment instructions.
    e. Log band assignment.

STEP 3 — SEMANTIC DIFF
  Module:  CALIUSO-SEMANTIC-DIFF
  Actions:
    a. Run diff_checks DC1–DC4.
    b. For each semantic change: require declared_mapping.
    c. Trace downstream impact of all meaning changes.
    d. If SILENT_CHANGE detected: reject patch immediately.
       Route to CALIUSO-GOV-AUTHORITY.
    e. Log diff results.

STEP 4 — GOVERNANCE REVIEW
  Module:  CALIUSO-GOV-AUTHORITY → CALIUSO-SELF-GOV (if governance module)
  Actions:
    a. Submit change_request to CALIUSO-GOV-AUTHORITY.
    b. If governance module: meta_review required.
    c. Confirm no weakening of human protection, agency threshold, or
       anti-weaponization rules.
    d. Approve or reject. Log governance_record.

STEP 5 — VALIDATION
  Module:  CALIUSO-INFORMATION-GATE → CALIUSO-SEMANTIC-FREEZE
           → CALIUSO-BOUNDARY-MATRIX → CALIUSO-SUBSTRATE-VARS
  Actions:
    a. Re-run INFORMATION-GATE on the patch content itself.
    b. Re-run SEMANTIC-FREEZE with the new module included.
    c. Confirm freeze_state = PASS after new definitions are added.
    d. Re-run BOUNDARY-MATRIX: confirm the patch is in a safe zone.
    e. Confirm substrate state: humans have enough capacity to absorb
       this change. If not: defer to next DESIGN cycle.
    f. Log all validation results.

STEP 6 — DUAL-RUN (if Band B or C)
  Module:  CALIUSO-ACTIVATION-BAND-ENGINE (dual_run_protocol)
  Actions:
    a. Activate new and old versions in parallel.
    b. Run both for the required cycle count (N=1 Band B; N=2 Band C).
    c. Log outputs from both versions.
    d. Compare: equivalent or better → proceed to cutover.
    e. Degraded → rollback. Log reason.
    f. Cutover requires CALIUSO-GOV-AUTHORITY sign-off.

STEP 7 — CUTOVER / ROLLBACK + PATCH LOG
  Module:  CALIUSO-LEDGER → CALIUSO-714-SEAL
  Actions:
    a. If proceeding: deactivate old version. Mark old as DEPRECATED.
    b. Award CALIUSO 714 SEAL to new version (run all 9 seal conditions).
    c. Update CALIUSO-SEMANTIC-ROOT with any new bindings.
    d. Log patch in CALIUSO-LEDGER (full patch_log entry; see §12).
    e. Update CALIUSO-SEMANTIC-DIFF declared_mapping archive.
    f. Announce the change via CALIUSO-HUMAN-READABLE-SIGNAL.
    g. If rolling back: restore prior version, log reason, run COMPASS.

§8. FAILURE CASCADE MAP
════════════════════════════════════════════════════════════════════════════════

This section documents the known failure cascade patterns — how one module
failure propagates through the stack and what stops it at each layer.

────────────────────────────────────────────────────────────────────────────────
CASCADE-1: AGENCY THRESHOLD FAILURE
────────────────────────────────────────────────────────────────────────────────

Trigger:   Low-agency input enters the system.
Path:
  INFORMATION-GATE scores agency ≤ 20
  → AGENCY-THRESHOLD fires HARD-STOP-1
  → GATE-TO-BOUNDARY sets conservative_bias = TRUE
  → BOUNDARY-MATRIX floors zone at R1_GUARDED minimum
  → HARD-STOPS (HARD-STOP-1) blocks downstream action
  → CORR-TRIGGERS dispatches STABILIZE
  → LEDGER logs rejection
  → Human operator notified via HUMAN-READABLE-SIGNAL

Firebreak:  AGENCY-THRESHOLD hard stop. Cannot be bypassed by module logic.
Escalation: If human overrides repeatedly without justification:
            POWER-MONITOR detects pattern → escalates to GOV-AUTHORITY.

────────────────────────────────────────────────────────────────────────────────
CASCADE-2: SEMANTIC FREEZE FAILURE
────────────────────────────────────────────────────────────────────────────────

Trigger:   A critical term is undefined at session start.
Path:
  SEMANTIC-ROOT lookup fails for critical term
  → SEMANTIC-FREEZE reports freeze_state = FAIL
  → FREEZE-GATE closes (gate_state = CLOSED)
  → ALL DYNAMIC MODULES blocked
  → CORR-TRIGGERS dispatches STABILIZE
  → HARD-STOPS (HARD-STOP-2) fires
  → LEDGER logs freeze failure

Firebreak:  FREEZE-GATE blocks all dynamics until a new PASS freeze record
            exists. No module can run dynamics without this.
Resolution: Define the term in SEMANTIC-ROOT. Re-run SEMANTIC-FREEZE.
            Confirm freeze_state = PASS. FREEZE-GATE opens.

────────────────────────────────────────────────────────────────────────────────
CASCADE-3: HUMAN OVERLOAD
────────────────────────────────────────────────────────────────────────────────

Trigger:   Operator rho falls below 0.4 (DEPLETED or HIGH load).
Path:
  SUBSTRATE-VARS computes rho < 0.4 / within_safe_range = NO
  → HARD-STOPS (HARD-STOP-3) fires
  → MODE-ROUTER Rule 1 triggers CRISIS if unresolved
  → CAPACITY-LEDGER activates reset_protocol
  → LOAD-BALANCER blocks HIGH task assignments
  → RECOVERY-ENGINE dispatches recovery prescription
  → CORR-TRIGGERS dispatches REDUCE-SCOPE + PAUSE-HANDOFF
  → BOUNDARY-GUARD enforces protected zones

Firebreak:  HARD-STOP-3. No HIGH-demand tasks may be assigned to depleted
            operators regardless of mode or operational pressure.
Escalation: If same operator triggers recovery 3 consecutive cycles:
            GOV-AUTHORITY is notified for structural workload review.

────────────────────────────────────────────────────────────────────────────────
CASCADE-4: INCENTIVE CONCEALMENT
────────────────────────────────────────────────────────────────────────────────

Trigger:   A decision-maker has misalignment_score ≥ 3 and bad news
           is reported later than expected.
Path:
  POWER-MONITOR detects escalation_trigger
  → oversight_gate activated (independent reviewer required)
  → If confirmed concealment: CRISIS-CARD activated
  → CORR-TRIGGERS dispatches correction
  → COMPASS I5 (No Reward for Concealment) flags as violation
  → GOV-AUTHORITY receives structural change proposal
  → LEDGER logs all findings

Firebreak:  oversight_gate: an independent human reviewer must sign off
            before the action proceeds.
Escalation: Confirmed concealment → CRISIS unconditionally.

────────────────────────────────────────────────────────────────────────────────
CASCADE-5: CLARITY DRIFT (Rₜ WORSENING)
────────────────────────────────────────────────────────────────────────────────

Trigger:   ΔR_sign = WORSENED in the ledger (Rₜ₊₁ > Rₜ without DEBT mark).
Path:
  LEDGER entry records CLARITY-VIOLATION
  → CORR-TRIGGERS dispatches (VERIFY first; STABILIZE if persists)
  → DRIFT-LOCK checks DC1–DC5 for semantic contribution to drift
  → OBJECTIVE-ENGINE re-assesses ΔOrder
  → If 3 consecutive WORSENED entries: MODE-ROUTER reassesses mode
    (possible CRISIS if drift is compounding)

Firebreak:  MINIMAL-CORRECTION ladder. Starts at VERIFY; escalates only
            if lower rungs cannot resolve.
Escalation: 3 DEBT-OVERDUE entries or 3 consecutive WORSENED entries
            → CRISIS route.

────────────────────────────────────────────────────────────────────────────────
CASCADE-6: GOVERNANCE CAPTURE
────────────────────────────────────────────────────────────────────────────────

Trigger:   A governance module is changed without a meta-review, or
           safety thresholds are lowered without Band C governance.
Path:
  SELF-GOV detects change to governance module without meta_review record
  → SEMANTIC-DIFF detects threshold lowering (SILENT_CHANGE flag)
  → GOV-AUTHORITY receives the violation
  → Patch is rejected; ROLLBACK triggered
  → LEDGER logs governance violation
  → POWER-MONITOR alerted (misalignment_score adjustment)

Firebreak:  SELF-GOV automatic Band C classification for all governance
            module changes. SEMANTIC-DIFF SILENT_CHANGE detection.
Escalation: Confirmed governance capture → CRISIS-CARD.

────────────────────────────────────────────────────────────────────────────────
CASCADE-7: I42 REPEATED FAILURE
────────────────────────────────────────────────────────────────────────────────

Trigger:   ΔNeg_others < 0 for 3 consecutive cycles (system is consistently
           increasing entropy for others).
Path:
  OBJECTIVE-ENGINE Q4 fails 3 consecutive cycles
  → CORR-TRIGGERS dispatches REDUCE-SCOPE + PAUSE-HANDOFF
  → COMPASS I42 registers sustained violation
  → MODE-ROUTER re-evaluates (likely DRIFT or CRISIS)
  → If PAUSE-HANDOFF: human must redesign the objective
  → LEDGER logs all Q4 failures and handoff record

Firebreak:  PAUSE-HANDOFF as final fallback. Humans must redesign or stop.
Escalation: If harm is actively compounding: CRISIS-CARD.

§9. ANTI-PATTERNS CATALOG
════════════════════════════════════════════════════════════════════════════════

Anti-patterns are named failure behaviors that appear correct but are not.
This catalog provides recognition cues and specific countermeasures.

────────────────────────────────────────────────────────────────────────────────
AP-01: COMPLIANCE THEATER
────────────────────────────────────────────────────────────────────────────────
Description:
  All required fields are filled, all steps are nominally executed, all seals
  are present — but none of it changes behavior. The stack is decorative.
Recognition:
  - Protocols take less time than physically possible.
  - Ledger entries are identical cycle to cycle.
  - Corrections are never applied because triggers are never acknowledged.
  - No decision has ever been changed by a CALIUSO check.
Countermeasure:
  Dry-run validation (CALIUSO-MODULE-SCHEMA step 2) must produce a case where
  the protocol would have blocked or changed an action. If no such case exists
  in the operational history, the stack is theater.

────────────────────────────────────────────────────────────────────────────────
AP-02: SCOPE INFLATION BY ANALOGY
────────────────────────────────────────────────────────────────────────────────
Description:
  CALIUSO's authority is incrementally expanded by treating analogies as
  structural identities: "This is just like X, so we should apply CALIUSO
  to Y as well" — where the analogy has never been classified by
  CALIUSO-MAPPING-CLASSIFIER.
Recognition:
  - CALIUSO is being applied to a domain where it has never been explicitly
    authorized.
  - The justification is an analogy, not a structural proof.
  - No MAPPING-CLASSIFIER record exists for the proposed domain extension.
Countermeasure:
  Any domain extension must go through CALIUSO-MAPPING-CLASSIFIER and
  CALIUSO-GOV-AUTHORITY. Unclassified analogies are NEEDS_GOVERNANCE by default.

────────────────────────────────────────────────────────────────────────────────
AP-03: CORRECTION BYPASS VIA CRISIS
────────────────────────────────────────────────────────────────────────────────
Description:
  CRISIS mode is declared not because conditions warrant it, but to bypass
  normal invariant checks, speed up governance, or avoid accountability.
Recognition:
  - CRISIS is declared without citing a specific MODE-ROUTER rule.
  - The stabilization_lead is the same person who benefits from the bypass.
  - Exit criteria are vague or never checked.
  - CRISIS persists beyond its authorized time limit.
Countermeasure:
  Mandatory rule citation in crisis_declaration_record. Time-limited authority.
  Post-crisis audit. SELF-GOV reviews CRISIS-CARD activations for patterns.

────────────────────────────────────────────────────────────────────────────────
AP-04: LEDGER AS PUNISHMENT INSTRUMENT
────────────────────────────────────────────────────────────────────────────────
Description:
  CALIUSO-LEDGER records are used to build cases against individuals —
  surfacing their past errors, miscalculations, or corrections in contexts
  designed to embarrass, demote, or justify removal.
Recognition:
  - Ledger entries are referenced in performance reviews.
  - Corrections in the ledger are used as evidence of incompetence.
  - Individuals under-report corrections because of perceived career risk.
Countermeasure:
  Anti-weaponization clause (CALIUSO-LEDGER, CALIUSO-CORE-CONTRACT):
  Ledger records may not be used for individual punishment. Corrections are
  evidence of healthy functioning, not failure. Confirmed weaponization →
  CRISIS-CARD.

────────────────────────────────────────────────────────────────────────────────
AP-05: MYTHIC FRAMING CREEP
────────────────────────────────────────────────────────────────────────────────
Description:
  Over time, CALIUSO language accretes mystique: terms like "the Bone",
  "the Inner Engine", or "the Seal" begin to be treated as sacred or
  self-evidently correct — and are no longer questioned or scrutinized.
Recognition:
  - Proposals to modify a module are met with "you can't change that"
    without a governance-level justification.
  - Module names are used as arguments ("but that's what the Bone says").
  - The monolith is treated as a source of truth rather than a derived tool.
Countermeasure:
  CALIUSO-CORE-CONTRACT CI-8 (No Myth Dependence): every claim must be
  evaluable on its own merits. Module names are labels, not authority.
  All modules are subject to revision via the normal governance process.

────────────────────────────────────────────────────────────────────────────────
AP-06: DEBT LAUNDERING
────────────────────────────────────────────────────────────────────────────────
Description:
  Complexity expansions are routinely marked as DEBT with no intention of
  resolution, effectively making DEBT the normal mode of operation and
  defeating the purpose of the DEBT exception.
Recognition:
  - DEBT_REGISTRY has > 3 OPEN entries at any time (violating the cap).
  - DEBT resolution plans are boilerplate ("will address later").
  - No DEBT entry has ever been marked RESOLVED.
  - OVERDUE entries accumulate without triggering CORR.
Countermeasure:
  Hard DEBT cap: > 3 OPEN entries triggers CRISIS route. OVERDUE entries
  are CLARITY-VIOLATIONS. DEBT_REGISTRY is reviewed at every DESIGN cycle.

────────────────────────────────────────────────────────────────────────────────
AP-07: ZONE TOURISM
────────────────────────────────────────────────────────────────────────────────
Description:
  The applicability zone is used selectively: high-zone protections (R3) are
  invoked to avoid responsibility, while low-zone freedoms (R0) are invoked
  to avoid oversight — flipping zones depending on what is convenient.
Recognition:
  - The same type of action is classified in different zones across cycles
    without a documented reason for the change.
  - R3 is invoked for decisions where CALIUSO has clear domain legitimacy
    and sufficient epistemic quality.
  - R0 is invoked for actions that clearly have significant human impact.
Countermeasure:
  Zone classifications are logged with full axis scores. Any zone change
  for a repeated action type requires a SEMANTIC-DIFF-style declared_mapping:
  "In cycle N we used R0 because X. In cycle N+1 we used R3 because Y."

────────────────────────────────────────────────────────────────────────────────
AP-08: SILENT MODULE RETIREMENT
────────────────────────────────────────────────────────────────────────────────
Description:
  A module stops being used without being formally deprecated. It remains in
  the schema as ACTIVE but has not run in many cycles. New modules partially
  overlap with it, creating ambiguity about which is authoritative.
Recognition:
  - Module has no ledger entries for ≥ 5 cycles while status = ACTIVE.
  - Another module has been informally doing the same job.
  - Teams are unsure whether to follow the old or new module.
Countermeasure:
  Any module with no ledger activity for ≥ 5 cycles must be reviewed:
  either used again (ACTIVE confirmed), formally deprecated (DEP patch
  through PATCH-CLASSIFIER), or absorbed into another module (REV patch).

────────────────────────────────────────────────────────────────────────────────
AP-09: SIGNAL INFLATION
────────────────────────────────────────────────────────────────────────────────
Description:
  CALIUSO-HUMAN-READABLE-SIGNAL is used so frequently for low-stakes events
  that operators stop reading it — and miss critical HARD-STOP and CRISIS
  signals buried in the noise.
Recognition:
  - Signal cards are generated for every minor ledger entry.
  - Operators report that they "skim" or "skip" signal cards.
  - A HARD-STOP or CRISIS signal was missed because it looked like noise.
Countermeasure:
  Signal cards are tiered: INFO signals are batched (daily digest).
  CAUTION, CORRECTION, CRISIS, and HARD-STOP signals are immediate and
  distinct. Volume of INFO signals is reviewed at each DESIGN cycle.

────────────────────────────────────────────────────────────────────────────────
AP-10: HUMAN SUPREMACY EROSION
────────────────────────────────────────────────────────────────────────────────
Description:
  R3 protections are progressively narrowed through legitimate-seeming
  governance changes, each individually minor, until CALIUSO is effectively
  making life-affecting decisions without meaningful human oversight.
Recognition:
  - Axis 6 scoring rubric has been revised upward (fewer situations score ≤ 2).
  - The definition of "coercive power" has been narrowed.
  - Consent requirements have been relaxed through incremental governance.
  - Human override has become a formality rather than a genuine check.
Countermeasure:
  GOV-AUTHORITY cumulative delta_log tracks all changes to R3-related
  thresholds. Any cumulative change > 20% from the original 714 spec
  triggers a full safety review with independent meta-review.
  R3 absolute rule (§5.G.5) is a LOCKED invariant; it cannot be patched
  without replacing the entire stack (which restarts this same protection).

§13. COMPLETE GLOSSARY
════════════════════════════════════════════════════════════════════════════════

All terms are listed alphabetically with their canonical CALIUSO 714 definitions.
These definitions are the authoritative Σ for this monolith.

Σ (sigma)
  The set of all active term bindings at a given time t.
  Defined by CALIUSO-SEMANTIC-ROOT. Frozen at session start by
  CALIUSO-SEMANTIC-FREEZE.

A (Attractor set)
  A = { 0, 3, 6 } ⊂ Z₉. The stable triad of the inner engine.
  States in A receive more stationary mass under the transition matrix T.

agency
  The degree to which a text respects the reader's ability to independently
  evaluate and choose. Scored 0–100. Hard floor: 20.

agency_floor
  20. The minimum acceptable agency score. Inputs ≤ 20 trigger HARD-STOP-1.

alignment_gap_0to5
  Mismatch between current task requirements and operator values/skills/
  motivation. 0 = perfect fit; 5 = severe mismatch.

anti-weaponization
  The rule that no CALIUSO record, module, or output may be used as an
  instrument for punishing, coercing, or disadvantaging a human being.

applicability_zone
  The domain classification (R0–R3) that determines what CALIUSO may do
